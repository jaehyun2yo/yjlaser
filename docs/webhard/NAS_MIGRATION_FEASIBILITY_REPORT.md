## 2. 현재 아키텍처 분석

### 2.1 현행 파일 저장 흐름

```
[거래처 브라우저] ──HTTPS──→ [Next.js (Vercel)]
                                    │
                                    ▼
                            [NestJS API (Railway)]
                                    │
                          Presigned URL 발급
                                    │
                                    ▼
[거래처 브라우저] ──직접 PUT──→ [Cloudflare R2]
                                    │
                          확인 콜백 (confirm)
                                    │
                                    ▼
                            [NestJS API (Railway)]
                            메타데이터 → PostgreSQL
```

**핵심 특징:**

- **Presigned URL 직접 업/다운로드**: 파일 바이너리가 NestJS/Next.js 서버를 경유하지 않음
- **멀티파트 업로드**: 100MB 이상 파일은 10MB 청크 단위 분할 업로드
- **R2 경로 구조**: `webhard/company-{id}/{folderId}/{timestamp}-{random}-{filename}`
- **소프트 삭제**: 3일 휴지통 보관 후 영구 삭제

### 2.2 현행 구조의 장점

| 장점                 | 설명                                     |
| -------------------- | ---------------------------------------- |
| 글로벌 CDN 엣지      | Cloudflare 네트워크를 통한 빠른 다운로드 |
| 무제한 대역폭        | 사무실 회선과 무관하게 동작              |
| 서버리스 확장        | 트래픽 증가 시 자동 대응                 |
| 99.999999999% 내구성 | 11-nine 데이터 내구성                    |
| 운영 부담 제로       | 하드웨어 관리 불필요                     |

### 2.3 R2 현재 비용 (추정)

| 항목                | 단가          | 현재 사용량 | 월 비용               |
| ------------------- | ------------- | ----------- | --------------------- |
| 저장                | $0.015/GB/월  | 15GB        | $0.23                 |
| Class A 작업 (쓰기) | $4.50/백만 건 | ~1,000건/월 | ~$0.005               |
| Class B 작업 (읽기) | $0.36/백만 건 | ~5,000건/월 | ~$0.002               |
| 이그레스 (다운로드) | **무료**      | -           | $0                    |
| **월 합계**         |               |             | **~$0.24 (약 320원)** |

> R2의 최대 장점은 이그레스 무료. 현재 규모에서 월 비용은 사실상 무시 가능.

## 3. 기존 NAS(Synology DS215j) 부적합 판단

### 3.1 판단 근거 요약

| #   | 제약 사항               | 심각도     | 상세                                                                 | 회피 가능 여부          |
| --- | ----------------------- | ---------- | -------------------------------------------------------------------- | ----------------------- |
| 1   | Docker 미지원           | **치명적** | NestJS 컨테이너화 불가 (Synology 공식: x86_64 + 4GB RAM 이상만 지원) | 불가                    |
| 2   | ARM 32-bit CPU 아키텍처 | **치명적** | Node.js 20+, Prisma 6 쿼리엔진 모두 ARM32 공식 빌드 미제공           | 불가                    |
| 3   | 512MB RAM (확장 불가)   | **치명적** | OS + NestJS + Prisma + Socket.IO + 파일 I/O 최소 합산 ~650MB 필요    | 불가 (납땜 온보드)      |
| 4   | DSM 6.2 EOL             | **높음**   | 2024년 이후 보안 패치 종료, 외부 노출 시 CVE 무방비                  | 불가 (DS215j 최종 버전) |
| 5   | 800MHz 단일코어         | **높음**   | HMAC-SHA256 서명 + TLS 핸드셰이크 병목, 동시 요청 처리 불가          | 불가                    |
| 6   | 2-bay 제한              | **중간**   | RAID 1 시 단일 디스크 용량, 이론적 최대 2TB 한도 제약                | 용량 관리로 일부 완화   |

> **#1~#3이 단독으로도 치명적**이며, 어떤 소프트웨어 우회로도 해결할 수 없는 **하드웨어·펌웨어 수준 제약**이다.

### 3.2 상세 기술 분석

#### 3.2.1 Docker 미지원 — 치명적

현재 NestJS 백엔드(`webhard-api/`)는 Docker 컨테이너로 Railway에 배포된다:

```dockerfile
# webhard-api/Dockerfile (실제 코드)
FROM node:20-alpine          # ← x86_64 Alpine Linux 기반
RUN corepack enable && corepack prepare pnpm@latest --activate
RUN npx prisma generate      # ← Prisma 쿼리 엔진 바이너리 생성
RUN pnpm build               # ← NestJS TypeScript 컴파일
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/main"]
```

NAS 전환 시 동일한 NestJS 파일 서비스를 NAS 위에서 Docker로 구동해야 하는데, DS215j는 **Docker(Container Manager) 패키지를 지원하지 않는다**.

- **Synology 공식 요건:** Docker는 x86_64 CPU + 최소 4GB RAM 모델만 지원
- **DS215j:** Marvell Armada 370 ARM 32-bit + 512MB RAM → 두 요건 **모두 미달**

**Node.js 직접 설치 우회도 불가능한 이유:**

| 검토 방법                               | 결과                                                                                |
| --------------------------------------- | ----------------------------------------------------------------------------------- |
| Node.js 20 ARM32 공식 바이너리          | **미제공** (Node.js 20부터 ARMv7l/ARM32 공식 빌드 중단)                             |
| Node.js 18 ARM32 바이너리 (마지막 지원) | 2025-04 EOL 도달, 보안 패치 종료                                                    |
| 소스에서 직접 컴파일                    | V8 엔진 컴파일에 최소 2GB RAM 필요 → 512MB로 **물리적 불가**                        |
| cross-compile (외부에서 빌드 후 복사)   | ARM32 + DSM 6.2 glibc 호환성 미보장, native addon(`@prisma/engines`) 별도 빌드 필요 |

#### 3.2.2 ARM 32-bit 아키텍처 비호환 — 치명적

현재 시스템의 핵심 의존성이 ARM 32-bit를 지원하지 않는다:

| 의존성            | 버전 (package.json 실제 값)              | ARM32 지원     | 근거                                                         |
| ----------------- | ---------------------------------------- | -------------- | ------------------------------------------------------------ |
| **Node.js**       | 20+ (Dockerfile: `node:20-alpine`)       | ❌             | v20부터 공식 ARM32 빌드 중단                                 |
| **Prisma Client** | `@prisma/client: ^6.1.0`                 | ❌             | Rust 기반 쿼리 엔진(`libquery_engine`) ARM32 바이너리 미제공 |
| **AWS SDK v3**    | `@aws-sdk/client-s3: ^3.700.0`           | ⚠️ 이론상 가능 | 순수 JS이나 Node.js 20+ 필수 API 사용                        |
| **TypeScript**    | `^5.7.2` (타겟: `@types/node: ^22.10.2`) | ⚠️             | Node 22 API 타겟 → Node 18 이하 런타임 호환성 미보장         |

**Prisma 쿼리 엔진 문제 (가장 치명적):**

Prisma Client는 Rust로 작성된 네이티브 쿼리 엔진 바이너리를 런타임에 로드한다. 현재 `schema.prisma`의 37개 모델(`WebhardFile`, `WebhardFolder`, `Company`, `CompanyStorage` 등)에 대한 모든 DB 접근이 이 엔진을 경유한다. Prisma 6는 `linux-arm64-openssl-3.0.x` 타겟을 제공하지만 이는 **ARM64(aarch64)** 전용이며, **ARM32(armv7l) 바이너리는 존재하지 않는다**.

→ Prisma가 구동되지 않으면 **모든 DB 작업(파일 메타데이터 CRUD, 저장 용량 조회, 거래처 인증)이 불가능**하다.

#### 3.2.3 512MB RAM 부족 — 치명적

실제 코드베이스 분석에 기반한 메모리 요구량 산출:

| 프로세스/구성요소           | 최소 메모리    | 산출 근거 (코드 증거)                                                      |
| --------------------------- | -------------- | -------------------------------------------------------------------------- |
| DSM 6.2 OS + 시스템 서비스  | ~200MB         | Synology 시스템 데몬 + syslog + 디스크 관리                                |
| Node.js 20 런타임 (V8 Heap) | ~120–180MB     | NestJS 10 부트스트랩 + DI 컨테이너 + 모듈 로딩                             |
| Prisma Client 쿼리 엔진     | ~60–120MB      | Rust 바이너리 로드 + 37 모델 스키마 캐시 + 커넥션 풀                       |
| AWS SDK S3 Client           | ~20–30MB       | `S3Client` 싱글턴 + `maxSockets: 50` HTTP 커넥션 풀 (`storage.service.ts`) |
| Socket.IO 서버              | ~15–30MB       | WebSocket 업그레이드 + 연결 상태 유지 (`socket.io: ^4.8.3`)                |
| 멀티파트 업로드 버퍼        | ~40–80MB       | 10MB 청크 x 4 동시 업로드 (`multipart.ts: MAX_CONCURRENT_UPLOADS = 4`)     |
| 인메모리 캐시               | ~10–20MB       | `cache-manager` 저장 용량/성능 메트릭 캐시                                 |
| V8 GC 여유 공간             | ~30–50MB       | GC 동작을 위한 최소 힙 마진                                                |
| **합계**                    | **~495–710MB** |                                                                            |

**DS215j 가용 메모리: 512MB 고정 (온보드 납땜, 확장 슬롯 없음)**

- OS만 부팅해도 ~200MB 소모 → **애플리케이션 가용 ~310MB**
- NestJS + Prisma 최소 로딩만으로 ~180–300MB → **멀티파트 업로드, Socket.IO 동시 운용 불가**
- OOM Killer에 의한 프로세스 강제 종료가 반복 발생할 것으로 예상

#### 3.2.4 CPU 성능 병목 — 높음

800MHz Marvell Armada 370 (ARM 32-bit, 단일코어)에서의 연산 성능 분석:

| 작업                              | 예상 소요 시간 | x86_64 (J4125) 대비 | 코드 내 해당 위치                                        |
| --------------------------------- | -------------- | ------------------- | -------------------------------------------------------- |
| HMAC-SHA256 Presigned URL 서명    | ~50–80ms/건    | **10~15배 느림**    | `storage.service.ts: getUploadPresignedUrl()`            |
| TLS 1.3 핸드셰이크                | ~200–300ms/건  | **8~10배 느림**     | 외부 HTTPS 접속 시 매 연결                               |
| Prisma 쿼리 직렬화/역직렬화       | ~30–50ms/건    | **5~8배 느림**      | `storage.service.ts: getStorageBreakdown()` 등 복합 쿼리 |
| 멀티파트 배치 Presigned URL (N개) | ~50ms x N      | 직렬 처리           | `multipart.ts: generatePartUploadUrls()` — CPU 바운드    |

**실사용 시나리오 예측:**

```
[시나리오] 거래처 A가 DXF 5개(각 2MB)를 업로드
  1. TLS 핸드셰이크:         ~250ms
  2. 세션 인증 (Prisma 쿼리): ~100ms
  3. 저장 용량 확인:          ~80ms   ← getStorageUsage()
  4. Presigned URL 5건 생성:  ~400ms  ← HMAC-SHA256 x 5
  5. 업로드 확인 콜백 5건:    ~250ms  ← DB 메타데이터 INSERT x 5
  합계:                      ~1,080ms (API 응답만, 파일 전송 제외)

  → x86_64 NAS에서 동일 작업: ~120ms
```

동시 접속 시 심화:

- 단일코어이므로 모든 요청이 **순차 처리** (Node.js 이벤트 루프 CPU 바운드 작업 직렬화)
- 피크 시 5명 동시 접속 시, 마지막 사용자는 **~5초 대기** 예상
- AES-NI 하드웨어 가속 미지원 → TLS 모든 연산이 소프트웨어 처리

#### 3.2.5 DSM 6.2 보안 EOL — 높음

| 항목                        | 상태                                         |
| --------------------------- | -------------------------------------------- |
| DSM 6.2 최종 보안 업데이트  | 2024년 종료                                  |
| DS215j의 DSM 7.x 업그레이드 | **불가** (Synology 공식 미지원 목록)         |
| OpenSSL 버전                | DSM 6.2 기본 OpenSSL 1.0.x → 알려진 CVE 다수 |
| Let's Encrypt 인증서        | ACME v2 클라이언트 호환성 미보장             |

외부 인터넷(거래처)에서 접근 가능한 파일 서버를 **보안 패치가 종료된 OS**에서 운영하는 것은:

- 거래처 도면(영업 비밀) 유출 위험
- 랜섬웨어 감염 위험 (NAS 대상 랜섬웨어 다수 존재)
- B2B 거래처 신뢰도 하락 위험

#### 3.2.6 저장 용량 제한 — 중간

| 구성                   | 최대 용량 | 이론적 최대 수요 대비          |
| ---------------------- | --------- | ------------------------------ |
| 2-bay RAID 1 (4TB x 2) | 4TB       | 200% 충분 (이론적 최대 ~2.1TB) |
| 2-bay RAID 0 (4TB x 2) | 8TB       | 충분하나 **데이터 보호 없음**  |
| 2-bay JBOD             | 8TB       | 디스크 독립, 장애 시 절반 손실 |

현재 15GB 기준으로는 용량 자체는 문제 없으나, 2-bay는 **RAID 5/SHR-1 구성이 불가능**하여 용량과 안전성을 동시에 확보할 수 없다.

### 3.3 종합 판정

```
DS215j 적합성 체크리스트:
  [ ] Docker 지원           → 미지원 (하드웨어 제약)
  [ ] Node.js 20+ 구동      → ARM32 바이너리 없음
  [ ] Prisma 6 쿼리엔진     → ARM32 네이티브 바이너리 없음
  [ ] 512MB 이상 RAM        → 512MB 고정, 확장 불가
  [ ] 보안 업데이트          → DSM 6.2 EOL, DSM 7 업그레이드 불가
  [ ] AES-NI (TLS 가속)     → 미지원
  [O] 저장 용량             → 현재 충분, 장기 확장 제한
  결과: 7개 필수 항목 중 6개 미충족
```

> **최종 판정: Synology DS215j는 yjlaser 웹하드 파일 서버 용도로 사용이 불가능하다.**
>
> ARM 32-bit 아키텍처, 512MB 고정 RAM, Docker 미지원이라는 **하드웨어 수준의 제약**으로 인해 현재 시스템의 핵심 의존성(Node.js 20, Prisma 6, AWS SDK v3)을 구동할 수 없다. 소프트웨어 우회(크로스 컴파일, 경량 대안 등)로도 해결이 불가능하며, DSM 6.2 보안 지원 종료로 외부 노출 자체가 위험하다.
>
> **권장 활용:** 파일 서버 대신, 기존 R2 데이터의 **로컬 백업 미러**로 활용 (rclone/rsync 야간 동기화, 추가 투자 0원)

## 4. NAS 전환 시 최소 스펙 및 추천 모델

### 4.1 최소 요구 스펙

#### 4.1.1 Docker/컨테이너 구동 요구사항 — 필수(MUST)

NAS 전환의 **가장 핵심 전제**는 Docker 컨테이너를 구동할 수 있는가이다. 현재 NestJS 백엔드는 Docker로 Railway에 배포되며, NAS에서도 동일한 컨테이너 환경을 재현해야 한다.

**현재 Docker 구성 분석** (`webhard-api/Dockerfile`):

```dockerfile
FROM node:20-alpine          # x86_64 Alpine Linux, ~180MB 이미지
RUN corepack enable && corepack prepare pnpm@latest --activate
RUN pnpm install --frozen-lockfile   # 의존성 38개 패키지
RUN npx prisma generate              # Rust 기반 쿼리엔진 바이너리 생성
RUN pnpm build                       # NestJS TypeScript → JavaScript 컴파일
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/main"]
```

| Docker 요구사항              | 상세                              | 코드 증거                                                      |
| ---------------------------- | --------------------------------- | -------------------------------------------------------------- |
| **아키텍처**                 | x86_64 필수 (ARM64는 조건부 가능) | `FROM node:20-alpine` — 공식 이미지 x86_64                     |
| **Docker Engine**            | 20.10+ (containerd 런타임)        | Prisma 6 `@prisma/client: ^6.1.0` — 쿼리엔진 바이너리 로딩     |
| **Docker Compose**           | 방식 B(터널+MinIO) 선택 시 필수   | NestJS + MinIO + Cloudflare Tunnel 3개 컨테이너 오케스트레이션 |
| **컨테이너 레지스트리 접근** | Docker Hub pull 가능해야 함       | `node:20-alpine`, `minio/minio`, `cloudflare/cloudflared`      |
| **볼륨 마운트**              | 호스트 ↔ 컨테이너 데이터 영속화   | 업로드 파일 저장 경로, Prisma migration 파일                   |
| **네트워크 모드**            | bridge + 포트 포워딩              | NestJS `:4000`, MinIO `:9000/:9001`                            |

**NAS 벤더별 Docker 지원 현황:**

| 플랫폼                     | Docker 패키지명   | 최소 요건         | Docker Compose | 비고                                 |
| -------------------------- | ----------------- | ----------------- | -------------- | ------------------------------------ |
| **Synology DSM 7.2+**      | Container Manager | x86_64 + 4GB RAM  | ✅ YAML 임포트 | GUI + CLI 모두 지원                  |
| **QNAP QTS 5.1+**          | Container Station | x86_64 + 4GB RAM  | ✅             | LXC도 추가 지원                      |
| **Synology DSM 7.x ARM64** | Container Manager | aarch64 + 4GB RAM | ✅             | Prisma ARM64 빌드 존재하나 검증 필요 |
| Synology DSM 6.x           | Docker (레거시)   | x86_64 + 2GB RAM  | ⚠️ 제한적      | Compose v1만, 보안 EOL               |
| ARM 32-bit NAS             | **미지원**        | —                 | ❌             | DS215j 포함 전 기종 불가             |

> **판정:** Docker(Container Manager) 공식 지원 + x86_64 아키텍처가 **절대 필수**. ARM64 NAS는 Prisma 쿼리엔진 호환성 검증이 추가로 필요하므로 x86_64를 우선 권장한다.

#### 4.1.2 CPU 요구사항

| 항목              | 최소          | 권장            | 산출 근거 (코드 증거)                                                |
| ----------------- | ------------- | --------------- | -------------------------------------------------------------------- |
| **아키텍처**      | x86_64        | x86_64          | `Dockerfile: FROM node:20-alpine` — ARM32 바이너리 없음              |
| **코어 수**       | 2코어/4스레드 | 4코어/4스레드   | Node.js 이벤트 루프 + Prisma 쿼리엔진(별도 스레드) + TLS 처리 병렬화 |
| **클럭**          | 2.0GHz+       | 2.5GHz+         | HMAC-SHA256 서명 ~5ms/건 @2.5GHz vs ~15ms @1.0GHz                    |
| **AES-NI**        | 필수          | 필수            | TLS 1.3 핸드셰이크 — 외부 거래처 HTTPS 접속 시 매 연결 수행          |
| **하드웨어 가속** | AES-NI        | AES-NI + SHA-NI | Presigned URL 서명(`storage.service.ts: getUploadPresignedUrl()`)    |

**CPU 바운드 워크로드 분석:**

```
[워크로드 1] Presigned URL 서명 (HMAC-SHA256)
  코드: storage.service.ts → getUploadPresignedUrl(), getDownloadPresignedUrl()
  빈도: 파일 업/다운로드마다 1회, 멀티파트 시 파트 수만큼 반복
  부하: 10MB 청크 멀티파트 100MB 파일 → 10회 HMAC 서명 연속 처리

[워크로드 2] TLS 핸드셰이크 (ECDHE + AES-256-GCM)
  코드: 외부 거래처 HTTPS 접속 시 매 연결
  빈도: 피크 시 5~15명 동시 접속 (§1.3 전제)
  부하: AES-NI 없으면 소프트웨어 처리로 ~10배 지연

[워크로드 3] Prisma 쿼리엔진 (Rust 네이티브 바이너리)
  코드: 38개 모델 CRUD — WebhardFile, WebhardFolder, Company 등
  빈도: API 요청마다 1~N회 DB 쿼리
  부하: JSON 직렬화/역직렬화가 CPU 집약적 (대량 파일 목록 조회 시)

[워크로드 4] 동시 업로드 처리
  코드: uploadQueue.ts — CONCURRENT_UPLOADS = 8 (동시 파일 8개)
        multipart.ts — MAX_CONCURRENT_UPLOADS = 4 (파트 동시 4개)
  부하: 8 파일 × 4 파트 = 최대 32건 동시 I/O + HMAC 서명
```

> **최소 2코어** 이유: Node.js 메인 스레드(이벤트 루프) + Prisma 쿼리엔진(별도 스레드)이 **동시에** 작동해야 한다. 단일코어에서는 TLS + HMAC 서명이 이벤트 루프를 블로킹하여 API 응답 지연이 누적된다.

#### Tier 1: 최소 비용 (50~60만원대)

| 모델                | CPU                                 | RAM            | 베이  | Docker               | AES-NI | 참고가  |
| ------------------- | ----------------------------------- | -------------- | ----- | -------------------- | ------ | ------- |
| **Synology DS224+** | Intel Celeron J4125 (4C/4T, 2.7GHz) | 2GB (최대 6GB) | 2-bay | ✅ Container Manager | ✅     | ~55만원 |
| QNAP TS-233         | ARM Cortex-A55 4코어                | 2GB            | 2-bay | ⚠️ 제한적            | ❌     | ~25만원 |

#### Tier 2: 균형 (70~90만원대) ⭐ **최종 추천**

| 모델                                    | CPU                                     | RAM             | 베이  | Docker               | AES-NI | 참고가  |
| --------------------------------------- | --------------------------------------- | --------------- | ----- | -------------------- | ------ | ------- |
| **ASUSTOR AS5404T** (Lockerstor 4 Gen2) | Intel Celeron N5105 (4C/4T, 2.0~2.9GHz) | 4GB (최대 16GB) | 4-bay | ✅ Portainer (ADM)   | ✅     | ~55만원 |
| **Synology DS923+**                     | AMD Ryzen R1600 (2C/4T, 2.6GHz)         | 4GB (최대 32GB) | 4-bay | ✅ Container Manager | ✅     | ~85만원 |
| QNAP TS-464                             | Intel Celeron N5095 (4C/4T, 2.9GHz)     | 8GB (최대 16GB) | 4-bay | ✅ Container Station | ✅     | ~80만원 |

> **DS923+ 최종 추천.** Docker/컨테이너 관점에서의 구체적 이유:
>
> 1. **4GB 기본 RAM → 즉시 Docker 구동 가능** — Container Manager 최소 요건 충족, 추가 구매 없이 NestJS 컨테이너 즉시 배포
> 2. **최대 32GB 확장** — 방식 B(NestJS + MinIO + Tunnel) 3개 컨테이너 동시 구동에도 충분한 여유. DDR4 ECC SO-DIMM 지원으로 안정성 확보
> 3. **AMD Ryzen R1600 — AES-NI + SHA 하드웨어 가속** — `storage.service.ts`의 Presigned URL HMAC-SHA256 서명과 TLS 1.3 핸드셰이크를 하드웨어로 처리. DS215j 대비 **10~15배 빠른 암호화 성능**
> 4. **4-bay → SHR-1 RAID** — RAID 5(3+1) 또는 SHR-1 구성으로 2.1TB 이론적 최대 수요 대응 + 1디스크 장애 허용
> 5. **10GbE NIC 확장 슬롯 (PCIe 3.0 x1)** — 사내 LAN 구간 고속화 대비
> 6. **Docker Compose YAML 임포트** — DSM 7.2 Container Manager에서 `docker-compose.yml` 파일을 GUI로 직접 배포 가능

**DS923+ Docker 구동 시뮬레이션:**

```
[구성] DSM 7.2 + Container Manager
  컨테이너 1: NestJS (node:20-alpine)
    - CPU: 2코어 제한
    - RAM: 512MB 제한 (피크 시 1GB까지 버스트)
    - 포트: 4000:4000
    - 볼륨: /volume1/webhard → /app/uploads

  컨테이너 2: MinIO (방식 B 선택 시)
    - CPU: 1코어 제한
    - RAM: 512MB 제한
    - 포트: 9000:9000, 9001:9001
    - 볼륨: /volume1/minio-data → /data

  컨테이너 3: Cloudflare Tunnel (방식 B 선택 시)
    - CPU: 0.5코어 제한
    - RAM: 128MB 제한
    - 네트워크: host 모드

  시스템 합계: ~1.5GB RAM 사용 (4GB 중), CPU 부하 평균 15~25%
  → 정상 운용 범위, 여유 확보됨
```

## 5. 아키텍처 방식 비교

NAS 전환 시 두 가지 아키텍처를 검토한다.

### 5.1 방식 A: 온프레미스 완전 이전

```
[거래처 브라우저] ──HTTPS──→ [Cloudflare Tunnel / Reverse Proxy]
                                        │
                                        ▼
                                [NAS (NestJS Docker)]
                                   │          │
                              파일 I/O    PostgreSQL
                              (로컬 SSD)   (Docker)

[사내 PC] ──LAN──→ [NAS (NestJS Docker)]
```

**설명:** NestJS 백엔드 + PostgreSQL + 파일 저장을 모두 NAS에서 운영. Railway/Vercel 의존도 제거.

| 장점                     | 단점                                       |
| ------------------------ | ------------------------------------------ |
| 데이터 완전 자체 보유    | NAS 장애 = 전체 서비스 중단                |
| 월 운영비 0원 (전기세만) | 100Mbps 회선이 **업/다운로드 병목**        |
| LAN 내부 초고속 접근     | 외부 접속 시 업로드 ~10MB/s 상한           |
| 단순한 구조              | 백업, 모니터링, SSL 갱신 직접 관리         |
|                          | PostgreSQL on NAS → 성능 제약              |
|                          | **Vercel → NAS API 호출 시 레이턴시 증가** |

**100Mbps 회선 병목 분석:**

| 시나리오                | 소요 시간 | 평가      |
| ----------------------- | --------- | --------- |
| 1MB DXF 다운로드 1건    | ~0.1초    | 쾌적      |
| 10MB PDF 다운로드 1건   | ~1초      | 양호      |
| 5명 동시 10MB 다운로드  | ~5초/건   | 수용 가능 |
| 10명 동시 10MB 다운로드 | ~10초/건  | 체감 느림 |
| 500MB ZIP 다운로드      | ~50초     | 불편      |

> **결론:** DXF/PDF 위주 + 낮은 동시접속이면 100Mbps로 충분. 단, 대용량 ZIP 일괄 다운로드 시 불편.

### 5.2 방식 B: 터널 연결 (하이브리드)

```
[거래처 브라우저] ──HTTPS──→ [Next.js (Vercel)]
                                    │
                                    ▼
                            [NestJS API (Railway)]
                                    │
                          Presigned URL 또는 Proxy
                                    │
                        ┌───────────┴───────────┐
                        ▼                       ▼
               [Cloudflare Tunnel]        [사내 LAN 직접]
                        │                       │
                        ▼                       ▼
                   [NAS 파일서버]          [NAS 파일서버]
```

**설명:** NestJS는 Railway에 유지. 파일 저장만 NAS로 이전. Cloudflare Tunnel로 NAS를 외부에 노출.

| 장점                             | 단점                                                |
| -------------------------------- | --------------------------------------------------- |
| NestJS/DB 안정성 유지 (Railway)  | Cloudflare Tunnel 의존성 추가                       |
| 점진적 마이그레이션 가능         | 터널 지연 추가 (~30–80ms)                           |
| Vercel ↔ Railway 기존 경로 유지  | 파일 프록시 구조 복잡도 증가                        |
| 외부 접속도 Cloudflare 보안 적용 | NAS 장애 시 파일 접근 불가 (API는 정상)             |
|                                  | Presigned URL → NAS 직접 접근 구조 변경 필요        |
|                                  | **Railway → 터널 → NAS 경로에서 100Mbps 병목 동일** |

**터널 성능 분석:**

- Cloudflare Tunnel 자체 오버헤드: ~10–30ms (국내 엣지 기준)
- Railway(미국 리전) → Cloudflare → 사무실 NAS: RTT ~150–250ms
- 파일 전송은 대역폭 위주이므로 RTT보다 **100Mbps 상한이 결정적**

### 5.3 방식 비교 매트릭스

#### 5.3.1 종합 비교표

| 평가 항목               | 방식 A (온프레미스 완전 이전)       | 방식 B (터널 하이브리드)             | 판정       |
| ----------------------- | ----------------------------------- | ------------------------------------ | ---------- |
| **구현 복잡도**         | 높음 — NestJS+DB+파일 전체 재배포   | 중간 — 파일 저장 계층만 변경         | **B 유리** |

| **마이그레이션 리스크** | 높음 — 빅뱅 전환, 롤백 어려움       | 낮음 — 점진적 전환, R2 폴백 가능     | **B 유리** |
| **운영 복잡도**         | 높음 — NAS가 SPOF, 전체 장애        | 중간 — 파일만 NAS 의존, API는 정상   | **B 유리** |
| **외부 다운로드 속도**  | 100Mbps 상한 (~10MB/s)              | 100Mbps 상한 (~10MB/s)               | **동등**   |
| **내부 LAN 속도**       | 1Gbps+ (최대 장점)                  | 1Gbps+ (동일 LAN 접근 가능)          | **동등**   |
| **월 운영 비용**        | 전기세만 (~1만원)                   | Railway($5~)+Vercel(무료)+전기세     | **A 유리** |

| **장애 복구**           | 어려움 — 자체 백업·복구 전부 필요   | 중간 — API는 클라우드, 파일만 복구   | **B 유리** |
| **보안 (외부 접근)**    | 자체 관리 (방화벽, SSL, DDoS)       | Cloudflare 보안 계층 활용            | **B 유리** |
| **확장성**              | NAS 하드웨어 한계 (물리 교체)       | API는 클라우드 확장, 파일만 NAS      | **B 유리** |
| **롤백 용이성**         | 어려움 — R2 복귀 시 전체 재구축     | 용이 — 스토리지 설정만 R2로 복원     | **B 유리** |
| **기존 코드 변경량**    | 많음 — 프론트+백엔드 전면 교체      | 적음 — MinIO 시 S3 SDK 호환 유지     | **B 유리** |
| **모니터링**            | 자체 구축 (Uptime, 디스크, 온도)    | Railway 모니터링 + NAS 파일만 감시   | **B 유리** |
| **SSL 인증서 관리**     | 자체 갱신 (Let's Encrypt cron)      | Cloudflare 자동 관리                 | **B 유리** |
| **Vercel↔API 레이턴시** | 증가 — Vercel→사무실NAS (RTT~150ms) | 유지 — Vercel→Railway (기존 경로)    | **B 유리** |

> **종합 판정: 14개 항목 중 방식 B 12건 유리, 방식 A 2건 유리, 동등 2건**
> 방식 A는 비용 절감과 완전한 데이터 주권에서만 우위. 나머지 모든 항목에서 방식 B가 유리하거나 동등.

### 5.4 Presigned URL 구조 변경 영향 (코드 기반 분석)

현재 R2 Presigned URL 직접 업/다운로드 구조는 NAS 전환 시 변경이 필요하다. 실제 코드 분석 결과:

#### 5.4.1 영향받는 파일 목록

| 계층          | 파일 경로                                      | 역할                          | 방식 A 변경       | 방식 B 변경                    |
| ------------- | ---------------------------------------------- | ----------------------------- | ----------------- | ------------------------------ |
| FE 클라이언트 | `src/lib/r2/client.ts`                         | S3Client 싱글톤               | 전면 교체         | **MinIO 시 엔드포인트만 변경** |
| FE 업로드     | `src/lib/r2/upload.ts`                         | Presigned PUT + 스트림 업로드 | 전면 교체         | **MinIO 시 최소 변경**         |

#### 5.4.2 MinIO 활용 시 코드 변경 최소화 (방식 B 핵심 전략)

NAS에 **MinIO**(S3 호환 오브젝트 스토리지)를 Docker로 구동하면, 기존 `@aws-sdk/client-s3` 코드를 **거의 그대로 사용** 가능:

| 현재 (R2)               | MinIO on NAS               | 변경 사항         |
| ----------------------- | -------------------------- | ----------------- |
| `endpoint: R2_ENDPOINT` | `endpoint: MINIO_ENDPOINT` | 환경변수 1개 변경 |
| `R2_ACCESS_KEY_ID`      | `MINIO_ACCESS_KEY`         | 환경변수 1개 변경 |
| `R2_SECRET_ACCESS_KEY`  | `MINIO_SECRET_KEY`         | 환경변수 1개 변경 |
| `R2_BUCKET_NAME`        | `MINIO_BUCKET_NAME`        | 환경변수 1개 변경 |
| S3 Presigned URL 서명   | **동일 (S3 호환)**         | 코드 변경 없음    |
| 멀티파트 업로드         | **동일 (S3 호환)**         | 코드 변경 없음    |
| DeleteObjects 배치 삭제 | **동일 (S3 호환)**         | 코드 변경 없음    |

> **결론:** 방식 B + MinIO 조합 시 **환경변수 4개 변경 + Cloudflare Tunnel 설정**만으로 전환 가능.
> 방식 A는 PostgreSQL 마이그레이션까지 포함하여 **8개 이상 파일 전면 교체 + DB 이전 작업**이 필요.

### 5.5 동일 체감 속도 충족 평가 (R2 기준 속도 비교)

현재 R2 Presigned URL 직접 전송 속도를 **기준선(Baseline)**으로, 각 아키텍처 옵션이 동일한 사용자 체감 속도를 충족하는지 평가한다.

#### 5.5.1 현행 R2 속도 기준선 (Baseline)

현재 파일 전송 경로는 **브라우저 ↔ Cloudflare R2 직접 통신** (Presigned URL)이며, NestJS/Next.js 서버를 경유하지 않는다.

| 측정 항목                  | R2 현행 값                                | 근거                                         |
| -------------------------- | ----------------------------------------- | -------------------------------------------- |
| **파일 전송 경로**         | 브라우저 → Cloudflare 엣지(한국 POP) → R2 | Presigned URL PUT/GET 직접 통신              |
| **엣지 레이턴시 (RTT)**    | **5~15ms**                                | 한국 내 Cloudflare POP (서울·부산)           |
| **가용 대역폭**            | **사실상 무제한**                         | Cloudflare CDN 대역폭, 사무실 회선과 독립    |
| **사무실 사용자 다운로드** | 100Mbps (~12.5MB/s)                       | 사무실 인터넷 회선이 병목 (R2→인터넷→사무실) |
| **사무실 사용자 업로드**   | 100Mbps (~12.5MB/s)                       | 사무실 인터넷 회선이 병목 (사무실→인터넷→R2) |
| **거래처 사용자 다운로드** | 거래처 ISP 속도 (보통 100~500Mbps)        | R2 측 대역폭 제한 없음, 거래처 ISP가 결정    |
| **거래처 사용자 업로드**   | 거래처 ISP 업로드 속도                    | R2 측 대역폭 제한 없음                       |
| **API 응답 (메타데이터)**  | **20~50ms**                               | Vercel → Railway, 양쪽 클라우드 내부 통신    |
| **Presigned URL 서명**     | **~5ms**                                  | Railway NestJS에서 `getSignedUrl()` 실행     |
| **멀티파트 업로드**        | 10MB 청크 × 4 병렬                        | `multipart.ts`: `MAX_CONCURRENT_UPLOADS = 4` |

> **핵심:** R2의 체감 속도를 결정하는 최대 요소는 **사용자 측 인터넷 회선 속도**이다. R2 자체는 Cloudflare 글로벌 CDN으로 대역폭 병목이 없으며, 사무실에서도 100Mbps 회선 속도 전체를 파일 전송에 활용할 수 있다.

#### 5.5.3 레이턴시 경로 분석 (요청~첫 바이트 수신)

파일 전송 시작까지의 **TTFB (Time to First Byte)** 비교:

| 경로 구간                             | R2 현행                      | 방식 A                           | 방식 B                           |
| ------------------------------------- | ---------------------------- | -------------------------------- | -------------------------------- |
| ① 브라우저 → API (Presigned URL 요청) | 20~50ms (Vercel→Railway)     | **150~250ms** (Vercel→사무실NAS) | 20~50ms (Vercel→Railway, 동일)   |
| ② Presigned URL 서명 처리             | ~5ms (Railway)               | ~5ms (NAS Docker)                | ~5ms (Railway)                   |
| ③ API → 브라우저 (URL 응답)           | 20~50ms                      | 150~250ms                        | 20~50ms                          |
| ④ 브라우저 → 스토리지 (파일 요청)     | **5~15ms** (Cloudflare 엣지) | 30~80ms (Tunnel) 또는 <1ms (LAN) | 30~80ms (Tunnel) 또는 <1ms (LAN) |
| ⑤ 스토리지 응답 시작 (TTFB)           | **~5ms** (R2 엣지 캐시)      | ~10ms (NAS 디스크 I/O)           | ~10ms (NAS 디스크 I/O)           |
| **총 TTFB (외부 사용자)**             | **55~125ms**                 | **345~595ms**                    | **85~195ms**                     |
| **총 TTFB (사내 사용자)**             | **55~125ms**                 | **<20ms**                        | **<20ms**                        |

> **레이턴시 판정:**
>
> - **방식 A 외부:** Vercel→NAS 구간에서 RTT가 3~5배 증가. API 메타데이터 응답이 체감적으로 느려짐
> - **방식 B 외부:** API 경로가 현행과 동일하여 TTFB 증가폭이 작음 (~30ms 터널 오버헤드만)
> - **양 방식 내부:** LAN 직접 접근으로 현행 R2 대비 **TTFB 대폭 개선**

#### 5.5.4 동시 접속 시 대역폭 포화 분석

현행 R2는 Cloudflare CDN이 사용자별로 독립된 대역폭을 제공하지만, NAS 전환 시 **사무실 100Mbps 회선을 모든 외부 접속자가 공유**한다.

| 동시 접속 수 (외부) | R2 현행 (1MB DXF 기준) | NAS 전환 시 (방식 A/B 동일) | 대역폭 포화율   |
| ------------------- | ---------------------- | --------------------------- | --------------- |
| 1명                 | ~0.03초                | ~0.1초                      | 8% (여유)       |
| 5명                 | 각 ~0.03초 (CDN 분산)  | 각 ~0.5초 (대역폭 분할)     | 40% (양호)      |
| 10명                | 각 ~0.03초             | 각 ~1초                     | 80% (**주의**)  |
| 15명 (피크)         | 각 ~0.03초             | 각 ~1.5초                   | 100% (**포화**) |

> **B2B 특성 반영:** 피크 동시 접속 5~15명 (§1.3 전제), 주 파일 유형 DXF(~1MB) 고려 시 **100Mbps 회선으로 10명까지 수용 가능**. 15명 피크 시 포화 가능하나 발생 빈도 극히 낮음.

#### 5.5.5 종합 판정: 동일 체감 속도 충족 여부

| 평가 항목                          | R2 현행 기준        | 방식 A 충족 여부                  | 방식 B 충족 여부                |
| ---------------------------------- | ------------------- | --------------------------------- | ------------------------------- |
| **사내 LAN 체감 속도**             | 100Mbps (~12.5MB/s) | ✅ **초과 달성** (1Gbps, 10배↑)   | ✅ **초과 달성** (1Gbps, 10배↑) |
| **외부 소형 파일 (DXF ~1MB)**      | ~0.03초             | ✅ 충족 (~0.1초, 체감 동일)       | ✅ 충족 (~0.12초, 체감 동일)    |
| **외부 중형 파일 (PDF ~10MB)**     | ~0.2초              | ⚠️ 저하 (~1초, 수용 가능)         | ⚠️ 저하 (~1.1초, 수용 가능)     |

| **외부 동시 다운로드 (5명)**       | 각 ~0.03초          | ⚠️ 저하 (각 ~0.5초)               | ⚠️ 저하 (각 ~0.5초)             |
| **외부 동시 다운로드 (15명 피크)** | 각 ~0.03초          | ❌ 미충족 (각 ~1.5초)             | ❌ 미충족 (각 ~1.5초)           |
| **API 응답 레이턴시 (TTFB)**       | 55~125ms            | ❌ 미충족 (345~595ms, **3~5배↑**) | ✅ 충족 (85~195ms, 유사)        |
| **멀티파트 업로드 (100MB)**        | ~2초 (CDN, 4 병렬)  | ⚠️ 저하 (~10초)                   | ⚠️ 저하 (~10초)                 |

**최종 판정 요약:**

| 구분                           | 방식 A           | 방식 B           | 비고                             |
| ------------------------------ | ---------------- | ---------------- | -------------------------------- |
| **사내 사용자**                | ✅ **체감 향상** | ✅ **체감 향상** | NAS 전환의 유일한 체감 이점      |
| **외부 — 일반 업무 (DXF/PDF)** | ⚠️ **근사 충족** | ⚠️ **근사 충족** | 1MB DXF 기준 체감 차이 미미      |
| **외부 — 대용량/동시접속**     | ❌ **미충족**    | ❌ **미충족**    | 100Mbps 회선 병목 — 양 방식 동일 |
| **API 응답 속도**              | ❌ **미충족**    | ✅ **충족**      | 방식 B만 기존 API 경로 유지      |

> **결론:**
>
> 1. **사내 사용자 체감 속도는 NAS 전환으로 10배 향상** — 이것이 전환의 유일한 속도 이점
> 2. **외부 거래처의 일반 업무(DXF 1MB, PDF 수 MB)는 100Mbps로도 체감 차이가 거의 없음** — "동일 체감 속도" 근사 충족
> 3. **50MB+ 대용량 파일이나 15명 동시 접속 시 외부 체감 속도 저하 불가피** — 단, B2B 특성상 발생 빈도 극히 낮음 (§1.2, §1.3 전제)
> 4. **방식 A는 API 응답 레이턴시까지 저하되어 전체 UX 열화**, 방식 B는 API 경로 유지로 메타데이터 응답 속도 동일

### 5.6 24/7 무중단 운영 가용성 분석

각 아키텍처 옵션이 **24시간 365일 무중단 운영**을 충족할 수 있는지 평가한다. 전원·네트워크·하드웨어 장애 시나리오별로 서비스 가용성을 분석하고, SLA(Service Level Agreement)를 비교한다.

#### 5.6.1 기준선: 현행 R2 구성 SLA

현재 사용 중인 클라우드 서비스의 공식 SLA:

| 서비스                      | 공식 SLA             | 연간 허용 다운타임 | 실적 가용성 (업계 평가)               |
| --------------------------- | -------------------- | ------------------ | ------------------------------------- |
| **Cloudflare R2**           | **99.9%**            | 8시간 46분/년      | ~99.95%+ (R2 2023~2025 주요 장애 0건) |
| **Railway** (NestJS 호스팅) | **99.9%** (Pro 플랜) | 8시간 46분/년      | ~99.9% (간헐적 배포 재시작 포함)      |
| **Vercel** (Next.js 호스팅) | **99.99%**           | 52분/년            | ~99.99% (글로벌 엣지 다중화)          |
| **현행 복합 SLA**           | **~99.88%**          | ~10시간 31분/년    | R2 × Railway × Vercel 연쇄 의존       |

> **현행 복합 SLA 산출:** 0.999 × 0.999 × 0.9999 ≈ 0.9980 (99.80%)
> 단, 각 서비스가 **독립 장애**이므로 실질 SLA는 이보다 높음. 파일 접근 불가와 API 접근 불가가 동시에 발생할 확률은 극히 낮음.

#### 5.6.2 아키텍처별 SLA 예상

| 구성요소                   | 현행 (R2)          | 방식 A (온프레미스)      | 방식 B (터널 하이브리드)     |
| -------------------------- | ------------------ | ------------------------ | ---------------------------- |
| **파일 스토리지**          | R2 99.9%           | NAS 자체 (~99.0~99.5%)   | NAS 자체 (~99.0~99.5%)       |
| **API 서버**               | Railway 99.9%      | NAS Docker (~99.0~99.5%) | Railway 99.9%                |
| **웹 프론트**              | Vercel 99.99%      | Vercel 99.99%            | Vercel 99.99%                |
| **DB (PostgreSQL)**        | Railway 포함 99.9% | NAS Docker (~99.0~99.5%) | Railway 포함 99.9%           |
| **네트워크 접근**          | 클라우드 CDN       | 사무실 ISP (~99.5%)      | 사무실 ISP + Tunnel (~99.3%) |
| **복합 SLA (전체 서비스)** | **~99.80%**        | **~97.5~98.5%**          | **~98.5~99.2%**              |
| **연간 예상 다운타임**     | **~10~17시간**     | **~130~220시간**         | **~70~130시간**              |

> **NAS 자체 가용성 ~99.0~99.5% 근거:**
>
> - 가정/소규모 NAS는 공식 SLA 미제공 (Synology·QNAP 모두 소비자 제품)
> - 계획된 중단: DSM 업데이트 ~4회/년 × 15분 = 1시간, 디스크 점검/스크럽 ~월 1회 × 30분 = 6시간
> - 비계획 중단: 정전, ISP 장애, 하드웨어 이슈 → 연 10~40시간 예상 (아래 상세 분석)

#### 5.6.3 장애 시나리오별 가용성 분석

##### ① 전원 장애 (정전)

| 시나리오                          | 현행 (R2)    | 방식 A                                    | 방식 B                            |
| --------------------------------- | ------------ | ----------------------------------------- | --------------------------------- |
| **사무실 단시간 정전 (수분)**     | ✅ 영향 없음 | ❌ **전체 서비스 중단** (UPS 없으면 즉시) | ⚠️ **파일만 중단**, API 정상      |
| **사무실 장시간 정전 (수시간)**   | ✅ 영향 없음 | ❌ **전체 서비스 장기 중단**              | ⚠️ **파일만 장기 중단**, API 정상 |
| **빌딩/지역 정전 (수시간~1일)**   | ✅ 영향 없음 | ❌ **전체 서비스 불가**                   | ⚠️ **파일만 불가**, API 정상      |
| **UPS 보유 시 (APC BE425M 기준)** | —            | 10~15분 버퍼 (안전 종료만 가능)           | 10~15분 버퍼                      |

**전원 장애 상세 분석:**

- **DS923+ 소비 전력:** 유휴 32W, 부하 시 ~50W
- **APC BE425M (425VA/255W):** DS923+ 기준 약 **10~15분** 유지 → 안전 종료(graceful shutdown)만 가능, 운영 지속 불가
- **NAS 안전 종료 미수행 시 위험:** RAID 메타데이터 손상, Docker 컨테이너 파일시스템 불일치, PostgreSQL WAL 미플러시(방식 A) → **데이터 무결성 위험**
- **UPS 자동 종료 설정:** DSM에서 USB-HID UPS 자동 감지 → 배터리 부족 시 안전 종료 트리거 가능 (설정 필수)
- **복구 시간 (RTO):** 전원 복구 후 NAS 부팅 ~3~5분 + Docker 컨테이너 자동 시작 ~1~2분 = **총 5~7분**

> **판정:** 정전은 NAS 운영의 **가장 큰 무중단 운영 위협**이다. 클라우드(R2/Railway/Vercel)는 데이터센터 이중화 전원으로 정전 영향이 없다. 방식 A는 정전 시 전체 서비스가 중단되나, 방식 B는 파일 접근만 중단되고 API·메타데이터 조회는 정상 유지된다.

##### ② 네트워크 장애 (ISP/인터넷)

| 시나리오                               | 현행 (R2)                                  | 방식 A                                            | 방식 B                                                      |
| -------------------------------------- | ------------------------------------------ | ------------------------------------------------- | ----------------------------------------------------------- |
| **사무실 ISP 장애 (수분~수시간)**      | ✅ 거래처 접근 정상, 사무실만 R2 접근 불가 | ❌ **외부+내부 모두 중단** (API·DB·파일 전부 NAS) | ⚠️ **파일만 외부 중단**, 사내 LAN 파일 접근 가능, API 정상  |
| **Cloudflare 글로벌 장애**             | ⚠️ R2 접근 불가 (희소, 연 1~2회 수분)      | ⚠️ 터널/리버스프록시 의존 시 외부 접근 불가       | ⚠️ 터널 중단 → 외부 파일 접근 불가, API 정상                |
| **Railway 장애**                       | ⚠️ API 중단, 파일 Presigned URL 발급 불가  | ✅ 영향 없음 (NAS 자체 운영)                      | ⚠️ **API 중단**, 파일은 NAS에 있으나 API가 없으면 접근 불가 |
| **사내 LAN 장비 고장 (스위치/라우터)** | ✅ 거래처 정상, 사무실만 접근 불가         | ❌ 사무실 NAS 접근 불가 → 외부도 중단             | ⚠️ NAS 접근 불가 → 파일만 중단                              |
| **DNS 장애**                           | ⚠️ 도메인 해석 실패 (CloudflareDNS 이중화) | ⚠️ 동일                                           | ⚠️ 동일                                                     |

**네트워크 장애 심층 분석:**

- **사무실 ISP 가용성:** 국내 기업용 100Mbps 고정 회선 → SLA 약 **99.5~99.8%** (연 17~44시간 장애 예상)
- **ISP 장애 시 방식 A 치명적 이유:** NestJS API + PostgreSQL + 파일 저장이 모두 NAS에 있으므로 ISP 장애 = **거래처가 웹하드 자체를 사용할 수 없음** (메타데이터 조회, 폴더 탐색, 로그인 모두 불가)
- **ISP 장애 시 방식 B 제한적 영향:** API·DB는 Railway에서 정상 운영되므로 거래처는 로그인, 폴더 탐색, 메타데이터 조회 가능. 실제 파일 다운로드/업로드만 실패. **에러 메시지로 "파일 서버 일시 점검 중" 안내 가능**
- **Cloudflare Tunnel 재연결:** 터널 끊김 시 `cloudflared` 데몬이 자동 재연결 시도 (기본 30초 간격, 최대 5회 → ~2.5분 내 자동 복구). ISP 복구 시 수동 개입 불요

> **판정:** 네트워크 장애에서 방식 A는 **전체 서비스 SPOF**, 방식 B는 **파일만 영향**으로 피해 범위가 현저히 다르다. 방식 B는 ISP 장애 시에도 "파일 일시 불가" 수준의 부분 장애(graceful degradation)를 제공한다.

#### 5.6.4 RPO (Recovery Point Objective) 비교

데이터 손실 허용 범위를 아키텍처별로 평가한다.

| 데이터 유형               | 현행 (R2)                                 | 방식 A                                                                             | 방식 B                           |
| ------------------------- | ----------------------------------------- | ---------------------------------------------------------------------------------- | -------------------------------- |
| **파일 데이터 (DXF/PDF)** | R2 자체 3중 복제 → RPO ≈ 0                | RAID 1/SHR-1 → 단일 디스크 장애 시 RPO ≈ 0, 본체 고장 시 **마지막 외부 백업 시점** | RAID + R2 백업 병행 시 RPO ≈ 0   |
| **DB 메타데이터**         | Railway 자동 백업 (일 1회) → RPO ≤ 24시간 | NAS 내 pg_dump 스케줄 → RPO = 백업 주기                                            | Railway 자동 백업 → RPO ≤ 24시간 |
| **사용자 세션/캐시**      | Railway 메모리 → 휘발                     | NAS Docker 메모리 → 휘발                                                           | Railway 메모리 → 휘발            |

**백업 전략에 따른 RPO:**

| 백업 전략                         | 방식 A RPO                        | 방식 B RPO            |
| --------------------------------- | --------------------------------- | --------------------- |
| 백업 없음 (RAID만)                | ❌ NAS 본체 고장 시 **전체 손실** | ❌ 파일 전체 손실     |
| NAS → 외부 HDD 수동 백업 (주 1회) | 최대 7일 데이터 손실              | 최대 7일 파일 손실    |
| NAS → 클라우드 자동 백업 (일 1회) | 최대 24시간 손실                  | 최대 24시간 파일 손실 |
| NAS → R2 실시간 동기화 (rclone)   | RPO ≈ 동기 주기 (5~30분)          | RPO ≈ 동기 주기       |

> **권고:** 방식 B 선택 시 **R2를 백업 대상으로 유지**(NAS → R2 야간 동기화)하면 파일 RPO를 24시간 이내로 보장할 수 있다. 이 구성은 현행 R2 데이터를 그대로 보존하므로 롤백도 즉시 가능.

#### 5.6.5 장애 영향 범위 종합 비교 (Blast Radius)

```

│ ⑥ Cloudflare 장애     │ □│ □ │ ■  │ □│ □ │ ■  │ □│ □ │ ■  │
│ ⑦ Vercel 장애         │ ■│ □ │ □  │ ■│ □ │ □  │ ■│ □ │ □  │
```

> **핵심 발견:**
>
> - **현행 R2:** 10개 시나리오 중 ■ 중단 2건, **전체 서비스 동시 중단 0건** (각 장애가 서로 다른 계층에 격리)
> - **방식 A:** ■ 중단 16건, **전체 서비스 동시 중단 7건** — NAS가 API+DB+파일 모든 계층의 SPOF로 작용
> - **방식 B:** ■ 중단 8건, **전체 서비스 동시 중단 0건** — NAS 장애 시에도 API·웹은 정상 유지 (graceful degradation)

| **하드웨어 장애 내성**         | ✅ 클라우드 자체 이중화   | ❌ SPOF (본체 고장 시 수일 중단)     | ⚠️ 파일만 영향               |
| **Graceful Degradation**       | ⚠️ R2 장애 시 파일만 불가 | ❌ 부분 장애 불가능 (올오어낫싱)     | ✅ 파일 불가 시에도 API 정상 |
| **RTO (복구 시간)**            | 수분 (클라우드 자동)      | 수분~수일 (장애 유형별)              | 수분~수일 (파일만)           |
| **RPO (데이터 손실)**          | ≈0 (3중 복제)             | RAID 보호 + 백업 정책 의존           | RAID + R2 백업 시 ≈0         |
| **24/7 무중단 충족 여부**      | ✅ **충족**               | ❌ **미충족**                        | ⚠️ **조건부 충족**           |

**24/7 무중단 운영 최종 판정:**

1. **현행 R2:** 24/7 무중단 운영을 **충족**한다. 클라우드 서비스의 데이터센터 이중화 전원, 자동 복구, 글로벌 CDN 분산으로 사무실 환경 장애의 영향을 받지 않는다.

2. **방식 A (온프레미스):** 24/7 무중단 운영을 **충족하지 못한다.** NAS가 모든 서비스(API+DB+파일)의 단일 장애점(SPOF)이 되어, 정전·ISP 장애·하드웨어 고장 중 어느 하나라도 발생하면 **전체 서비스가 중단**된다. 소규모 NAS에는 이중화 구성(HA 클러스터)이 불가능하므로 구조적으로 해결할 수 없다.

3. **방식 B (터널 하이브리드):** 24/7 무중단 운영을 **조건부로 충족**한다. NAS 장애 시 파일 접근만 불가하고 API·웹·메타데이터 조회는 정상 유지되므로 **graceful degradation**(우아한 성능 저하)을 제공한다. 단, 파일 업로드/다운로드가 핵심 기능인 웹하드 특성상 "파일 접근 불가 = 서비스 불가"로 해석할 수도 있어, **UPS + R2 폴백 전략**을 병행해야 실질적 무중단에 근접한다.

> **방식 B + R2 폴백 전략:** NAS 장애 감지 시 자동으로 R2 경로로 전환하는 헬스체크 기반 폴백을 구현하면, NAS 다운타임 동안에도 파일 접근이 가능하다. 이 경우 방식 B의 실질 가용률은 **현행 R2 수준(99.8%+)**에 근접할 수 있다. 단, 폴백 구현에 추가 개발(1~2주)이 필요하며, NAS↔R2 간 파일 동기화 정합성 관리가 운영 복잡도를 높인다.

### 5.7 무중단 마이그레이션 전략 (R2 → NAS 전환)

§5.6이 **전환 후 운영 중 무중단**을 평가했다면, 이 절은 **전환 과정 자체의 무중단** — R2에서 NAS로 데이터를 이전하는 동안 기존 서비스를 중단 없이 유지할 수 있는지를 평가한다.

#### 5.7.1 마이그레이션 전제 조건 분석

| 항목                | 현황                                                                        | 마이그레이션 영향                                        |
| ------------------- | --------------------------------------------------------------------------- | -------------------------------------------------------- |
| **총 데이터량**     | ~15GB (DXF/PDF)                                                             | 100Mbps(~10MB/s)로 **~25분** 전송 가능                   |
| **파일 수**         | 수천 개 (평균 수십KB~수MB)                                                  | 소량이므로 개별 파일 검증 가능                           |
| **파일 메타데이터** | PostgreSQL `webhard_files.path` 컬럼에 R2 오브젝트 키 저장                  | **키 유지 시 DB 수정 불요**, 키 변경 시 전체 UPDATE 필요 |
| **활성 업로드**     | 업무 시간 중 간헐적 (피크 5~15명)                                           | 이전 중 신규 업로드 처리 전략 필요                       |
| **S3 호환성**       | `@aws-sdk/client-s3` 사용 (FE/BE 양쪽)                                      | MinIO(S3 호환) 사용 시 **오브젝트 키 경로 100% 호환**    |
| **환경변수 구조**   | `R2_ENDPOINT`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME` | 환경변수 4개 교체로 엔드포인트 전환                      |

> **핵심 유리 조건:** 15GB 소량 데이터 + S3 호환 SDK + 오브젝트 키 보존 가능 → 무중단 마이그레이션 **기술적으로 가능**.

#### 5.7.2 아키텍처별 마이그레이션 전략 비교

##### 방식 A: 온프레미스 완전 이전 — 마이그레이션 전략

방식 A는 NestJS API + PostgreSQL + 파일 저장을 모두 NAS로 이전하므로, 마이그레이션 범위가 크고 복잡하다.

**이전 대상:**

1. 파일 데이터: R2 → NAS 로컬 디스크 (또는 NAS Docker MinIO)
2. NestJS 백엔드: Railway → NAS Docker
3. PostgreSQL DB: Railway → NAS Docker
4. 프론트엔드 API 엔드포인트: `NEXT_PUBLIC_WEBHARD_API_URL` 변경

**마이그레이션 단계:**

```
Phase 0: 사전 준비 (1~2일)
├── NAS 초기 설정 (DSM, Docker, 네트워크)
├── Docker에 NestJS + PostgreSQL + MinIO 구동
├── Cloudflare Tunnel 설정 (외부 접속용)
└── NAS 환경 검증 (API 헬스체크, DB 접속, MinIO 접속)

Phase 1: 데이터 동기화 — R2 사본 생성 (수시간, 서비스 무영향)
├── rclone sync R2 → NAS MinIO (오브젝트 키 보존)
│   └── 15GB / ~10MB/s ≈ 25분 (실제 오버헤드 포함 ~1시간)
├── pg_dump Railway DB → NAS PostgreSQL 복원
└── 동기화 완료 검증 (파일 수·용량·체크섬 비교)

Phase 2: 증분 동기화 + 정합성 확인 (수시간)
├── Phase 1 이후 신규/변경 파일 증분 동기화
├── DB 변경분 WAL 기반 또는 재 dump 동기화
└── R2 vs NAS 파일 목록 diff 검증

Phase 3: ⚠️ 서비스 전환 (계획된 중단 필요)
├── ❌ 점검 공지 게시 ("00:00~02:00 서비스 점검")
├── ❌ Railway NestJS 중지 → 신규 요청 차단
├── ❌ 최종 증분 동기화 (마지막 변경분)
├── ❌ Vercel 환경변수 변경: NEXT_PUBLIC_WEBHARD_API_URL → NAS 터널 주소
├── ❌ Vercel 재배포 (환경변수 반영)
├── ❌ DNS/터널 경로 검증
└── ❌ 서비스 재개 확인
```

**방식 A 무중단 마이그레이션 평가:**

| 평가 항목                | 판정                 | 근거                                                                                                          |
| ------------------------ | -------------------- | ------------------------------------------------------------------------------------------------------------- |
| **파일 데이터 이전**     | ✅ 무중단 가능       | R2 → NAS 복사는 백그라운드 수행, 서비스 무영향                                                                |
| **DB 이전**              | ❌ **중단 필요**     | PostgreSQL 마이그레이션 시 **쓰기 중단 시점** 불가피 (WAL 스트리밍 미지원 환경)                               |
| **API 엔드포인트 전환**  | ❌ **중단 필요**     | `NEXT_PUBLIC_WEBHARD_API_URL` 변경 → Vercel 재배포 → 전파 지연 (~1~5분)                                       |
| **Presigned URL 연속성** | ❌ **기존 URL 무효** | 전환 전 발급된 R2 Presigned URL은 만료 전이라도 R2를 가리킴 → 전환 후 무효. 최대 1시간 내 자연 만료 대기 필요 |
| **DNS/라우팅 전환**      | ⚠️ 지연 발생         | Cloudflare Tunnel 경유 시 DNS 변경 불요하나, Vercel 재배포 전파 ~1~5분                                        |
| **종합 판정**            | ❌ **무중단 불가**   | DB + API 엔드포인트 동시 전환으로 **최소 30분~2시간 계획 중단** 불가피                                        |

> **방식 A 최소 중단 시간 산출:**
>
> - DB 최종 동기화: ~10~20분 (15GB DB 기준 수분, 안전 마진 포함)
> - Vercel 환경변수 변경 + 재배포: ~3~5분
> - 라우팅 검증 + 스모크 테스트: ~5~10분
> - Presigned URL 자연 만료 대기: 최대 60분 (다운로드 5분, 업로드 10분, 멀티파트 1시간)
> - **총 최소 계획 중단: ~30분~2시간** (야간 점검 권장)

##### 방식 B: 터널 하이브리드 — 마이그레이션 전략

방식 B는 NestJS API + PostgreSQL을 Railway에 유지하고 **파일 저장만** NAS로 이전하므로, 마이그레이션 범위가 좁고 단순하다.

**이전 대상:**

1. 파일 데이터만: R2 → NAS MinIO (S3 호환)
2. 환경변수: `R2_*` → `MINIO_*` (Railway 환경변수 4개)
3. Cloudflare Tunnel: NAS MinIO 외부 노출

**NestJS 코드 무변경 근거 (코드 분석 기반):**

| 코드 파일                                    | 현재 동작                                   | MinIO 전환 시                            | 변경 여부      |
| -------------------------------------------- | ------------------------------------------- | ---------------------------------------- | -------------- |
| `webhard-api/src/storage/storage.service.ts` | `S3Client({ endpoint: R2_ENDPOINT })`       | `S3Client({ endpoint: MINIO_ENDPOINT })` | **환경변수만** |
| `src/lib/r2/client.ts`                       | `S3Client({ endpoint: R2_ENDPOINT })`       | `S3Client({ endpoint: MINIO_ENDPOINT })` | **환경변수만** |
| `src/lib/r2/upload.ts`                       | `PutObjectCommand` + Presigned URL          | 동일 SDK 메서드                          | 변경 없음      |
| `src/lib/r2/download.ts`                     | `GetObjectCommand` + Presigned URL          | 동일 SDK 메서드                          | 변경 없음      |
| `src/lib/r2/multipart.ts`                    | `CreateMultipartUploadCommand` 등           | 동일 SDK 메서드                          | 변경 없음      |
| `src/lib/r2/delete.ts`                       | `DeleteObjectsCommand`                      | 동일 SDK 메서드                          | 변경 없음      |
| DB `webhard_files.path`                      | `webhard/company-{id}/...` 형태 오브젝트 키 | **동일 키** MinIO에 보존                 | 변경 없음      |

**마이그레이션 단계 (무중단 가능):**

```
Phase 0: 사전 준비 (반나절)
├── NAS에 MinIO Docker 컨테이너 구동
├── Cloudflare Tunnel 설정 → MinIO API 포트 외부 노출
├── MinIO 버킷 생성 (R2 버킷명과 동일: 'yjlaser')
└── MinIO 헬스체크 확인

Phase 1: 전체 데이터 복제 (R2 → NAS, 서비스 무영향)
├── rclone sync r2:yjlaser minio:yjlaser --progress
│   └── 15GB / ~10MB/s ≈ 25분 (R2 → 인터넷 → 사무실 → NAS)
├── 오브젝트 키 경로 완전 보존 (webhard/company-*/folder-*/...)
└── 파일 수·용량·체크섬 검증 (rclone check)

Phase 2: Dual-Write 활성화 (서비스 무영향)
├── NestJS StorageService에 Dual-Write 모드 추가:
│   ├── 업로드: R2 + NAS MinIO 양쪽에 동시 기록
│   ├── 다운로드: R2에서 우선 서빙 (기존 동작 유지)
│   └── 삭제: R2 + NAS MinIO 양쪽에서 동시 삭제
├── Railway 환경변수에 MINIO_* 추가 (R2_*는 유지)
├── Railway 배포 (Dual-Write 코드 + MinIO 환경변수)
└── Dual-Write 동작 검증 (신규 파일이 양쪽에 존재하는지 확인)

Phase 3: 증분 동기화 (Phase 1~2 사이 신규 파일 보완)
├── rclone sync --update (변경분만 동기화)
├── R2 vs NAS 오브젝트 키 전수 비교 (rclone check)
└── 불일치 0건 확인

Phase 4: 읽기 경로 전환 (무중단)
├── NestJS StorageService 읽기 경로를 NAS MinIO로 전환:
│   ├── Presigned URL 서명 → MINIO_ENDPOINT 사용
│   ├── 다운로드: NAS MinIO에서 서빙
│   └── R2는 백업 폴백으로 유지
├── 환경변수 전환: 기본 스토리지 = NAS MinIO
├── Railway 배포 (롤링 업데이트, 무중단)
└── 전환 확인: 새 Presigned URL이 NAS MinIO를 가리키는지 검증

Phase 5: R2 쓰기 중단 + 정리 (무중단)
├── Dual-Write에서 R2 쓰기 제거 (NAS MinIO 단독 쓰기)
├── Railway 배포 (롤링 업데이트)
├── R2 데이터는 백업 목적으로 30~90일 유지
└── 안정화 확인 후 R2 버킷 삭제 (선택)
```

**방식 B 무중단 마이그레이션 평가:**

| 평가 항목                | 판정               | 근거                                                                                                              |
| ------------------------ | ------------------ | ----------------------------------------------------------------------------------------------------------------- |
| **파일 데이터 이전**     | ✅ 무중단          | rclone 백그라운드 복사, 서비스 무영향                                                                             |
| **DB 이전**              | ✅ 해당 없음       | DB는 Railway에 유지, 이전 불요                                                                                    |
| **API 엔드포인트 전환**  | ✅ 무중단          | `NEXT_PUBLIC_WEBHARD_API_URL` 변경 없음 (Railway 유지)                                                            |
| **Presigned URL 연속성** | ✅ 무중단          | Dual-Write 기간 중 R2 URL은 유효. Phase 4에서 새 URL만 NAS로 전환. 기발급 R2 URL은 만료(최대 1시간)까지 자연 소멸 |
| **Dual-Write 신뢰성**    | ⚠️ 구현 필요       | `StorageService`에 Dual-Write 로직 추가 (~50줄). NAS 쓰기 실패 시 R2만 성공으로 처리 (비동기 재시도)              |
| **코드 변경 규모**       | ✅ 최소            | 환경변수 4개 + StorageService Dual-Write 임시 코드 (~50줄)                                                        |
| **종합 판정**            | ✅ **무중단 가능** | 서비스 중단 없이 5단계 점진적 전환 가능. 각 Phase 독립 실행·검증·롤백 가능                                        |

> **방식 B 전환 총 소요 시간 (중단 없음):**
>
> - Phase 0 (사전 준비): ~반나절
> - Phase 1 (전체 복제): ~1시간
> - Phase 2 (Dual-Write): ~1시간 (배포 포함)
> - Phase 3 (증분 동기화): ~30분
> - Phase 4 (읽기 전환): ~30분 (배포 포함)
> - Phase 5 (R2 정리): 안정화 후 30~90일 뒤
> - **총 마이그레이션 작업: ~1일 (서비스 중단 0분)**

#### 5.7.3 롤백 전략 비교

마이그레이션 중 문제 발생 시 즉시 원래 상태(R2)로 복귀할 수 있는지 평가한다.

##### 방식 A 롤백 전략

```
[문제 발생] → 롤백 필요
     │
     ▼
┌─ Phase 3 전환 전 (Phase 0~2) ─────────────────────────┐
│  ✅ 즉시 롤백 가능                                      │
│  • R2 + Railway가 그대로 운영 중                        │
│  • NAS 구성을 폐기하면 끝                               │
│  • 롤백 소요: 0분 (아무것도 안 바뀌었으므로)            │
└────────────────────────────────────────────────────────┘

┌─ Phase 3 전환 후 ─────────────────────────────────────┐
│  ⚠️ 복잡한 롤백                                        │
│  • NAS 전환 후 신규 업로드된 파일은 NAS에만 존재       │
│  • 롤백 시 NAS 신규 파일 → R2 역동기화 필요            │
│  • Vercel 환경변수 재변경 + 재배포 (~3~5분)            │
│  • Railway NestJS + PostgreSQL 재기동 (~5~10분)        │
│  • 전환 후 경과 시간이 길수록 롤백 데이터량 증가       │
│  • 롤백 소요: 30분~2시간 (데이터 역동기화 포함)        │
│  • ❌ 전환 후 DB 스키마 변경이 있었다면 롤백 매우 어려움│
└────────────────────────────────────────────────────────┘
```

| 롤백 시점                 | 난이도         | 소요 시간   | 데이터 손실 위험                  |
| ------------------------- | -------------- | ----------- | --------------------------------- |
| Phase 0~2 (전환 전)       | ✅ 즉시        | 0분         | 없음                              |
| Phase 3 직후 (수분 내)    | ⚠️ 중간        | ~30분       | 전환 후 업로드 파일 역동기화 필요 |
| Phase 3 후 수일 경과      | ❌ 어려움      | ~2~4시간    | 대량 역동기화 + DB 정합성 검증    |
| Phase 3 후 DB 스키마 변경 | ❌ 매우 어려움 | 수시간~수일 | DB 마이그레이션 역순 적용 필요    |

##### 방식 B 롤백 전략

```
[문제 발생] → 롤백 필요
     │
     ▼
┌─ 어느 Phase에서든 (Phase 5 전) ────────────────────────┐
│  ✅ 즉시 롤백 가능                                      │
│  • R2 데이터가 Phase 5 전까지 100% 보존되어 있음        │
│  • 환경변수 4개를 R2로 원복하면 즉시 복귀               │
│  • Railway 환경변수 변경 → 롤링 재시작 (~2분)          │
│  • DB 변경 없음 → DB 롤백 불요                         │
│  • Vercel 변경 없음 → 프론트엔드 영향 없음             │
│  • 롤백 소요: ~2~5분                                   │
└────────────────────────────────────────────────────────┘

┌─ Phase 5 이후 (R2 데이터 삭제 후) ─────────────────────┐
│  ⚠️ 역동기화 필요                                      │
│  • NAS → R2 역동기화 (rclone sync 역방향)              │
│  • 15GB 기준 ~1시간                                    │
│  • 환경변수 4개 R2로 원복                              │
│  • 롤백 소요: ~1~2시간                                 │
│  └ 권고: R2 데이터를 최소 90일 유지하여 이 시나리오 방지│
└────────────────────────────────────────────────────────┘
```

| 롤백 시점                 | 난이도      | 소요 시간 | 데이터 손실 위험                     |
| ------------------------- | ----------- | --------- | ------------------------------------ |
| Phase 0~1 (복제 중)       | ✅ 즉시     | 0분       | 없음 (R2 변경 없음)                  |
| Phase 2~3 (Dual-Write 중) | ✅ 즉시     | ~2분      | 없음 (R2에 모든 데이터 존재)         |
| Phase 4 (읽기 전환 후)    | ✅ 즉시     | ~2~5분    | 없음 (R2 데이터 보존 중)             |
| Phase 5 이후 (R2 삭제 후) | ⚠️ 역동기화 | ~1~2시간  | 동기화 간 업로드 파일은 NAS에만 존재 |

> **핵심 차이:** 방식 B는 R2 데이터를 삭제하지 않는 한 **어느 Phase에서든 2~5분 내 즉시 롤백**이 가능하다. 방식 A는 전환 후 DB까지 이전되었으므로 롤백 복잡도가 높다.

#### 5.7.4 Dual-Write 구현 상세 (방식 B 핵심 전략)

방식 B 무중단 마이그레이션의 핵심은 **Dual-Write 패턴**이다. 기존 `StorageService`에 최소한의 변경으로 구현 가능:

```
┌─────────────────────────────────────────────┐
│              StorageService                  │
│                                              │
│  Upload Request                              │
│       │                                      │
│       ├──→ [Primary: R2]  ← Phase 2~3       │
│       │         ✅ 성공 → 클라이언트 응답     │
│       │                                      │
│       └──→ [Secondary: NAS MinIO] (비동기)   │
│                 ✅ 성공 → 로그               │
│                 ❌ 실패 → 재시도 큐 + 알림   │
│                                              │
│  ─── Phase 4 전환 후 ───                     │
│                                              │
│       ├──→ [Primary: NAS MinIO]              │
│       │         ✅ 성공 → 클라이언트 응답     │
│       │                                      │
│       └──→ [Secondary: R2] (비동기, 백업)    │
│                 ✅/❌ → 로그만               │
└─────────────────────────────────────────────┘
```

**Dual-Write 설계 원칙:**

1. **Primary 우선 응답:** 클라이언트는 Primary 성공 시 즉시 응답 수신. Secondary 결과를 기다리지 않음
2. **Secondary 비동기:** Secondary 쓰기 실패가 사용자 경험에 영향 없음
3. **실패 복구:** Secondary 쓰기 실패 시 재시도 큐(간단한 메모리 큐 또는 Bull Queue)에 적재
4. **전환 플래그:** 환경변수 `STORAGE_PRIMARY=r2|minio`로 Primary/Secondary 전환 제어
5. **점진적 전환:** 트래픽 없는 야간에 `STORAGE_PRIMARY=minio`로 변경 → Railway 롤링 재시작

**구현 영향 범위:**

| 파일                                         | 변경 내용                                                   | 변경량                               |
| -------------------------------------------- | ----------------------------------------------------------- | ------------------------------------ |
| `webhard-api/src/storage/storage.service.ts` | Dual S3Client 인스턴스 + Primary/Secondary 분기             | ~50줄 추가                           |
| `.env.local`                                 | `MINIO_*` 환경변수 4개 + `STORAGE_PRIMARY` 플래그           | 5개 변수 추가                        |
| `src/lib/r2/client.ts`                       | `STORAGE_PRIMARY` 기반 엔드포인트 분기 (FE Presigned URL용) | ~10줄 수정                           |
| **합계**                                     |                                                             | **~60줄 임시 코드** (안정화 후 제거) |

#### 5.7.5 Presigned URL 전환 시 연속성 분석

무중단 마이그레이션에서 가장 주의할 점은 **이미 발급된 Presigned URL의 유효성**이다.

**현재 Presigned URL 만료 시간 (코드 분석):**

| 용도               | 만료 시간       | 소스 코드 참조                                              |
| ------------------ | --------------- | ----------------------------------------------------------- |
| 다운로드 (BE)      | 300초 (5분)     | `storage.service.ts` → `PRESIGNED_EXPIRES.DOWNLOAD = 300`   |
| 업로드 (BE)        | 600초 (10분)    | `storage.service.ts` → `PRESIGNED_EXPIRES.UPLOAD = 600`     |
| 멀티파트 파트 (BE) | 3,600초 (1시간) | `storage.service.ts` → `PRESIGNED_EXPIRES.MULTIPART = 3600` |
| 업로드 (FE)        | 3,600초 (1시간) | `upload.ts` → `PRESIGNED_URL_MAX_EXPIRY = 3600`             |
| 다운로드 (FE)      | 3,600초 (1시간) | `download.ts` → `getR2SignedUrl(key, 3600)`                 |

**전환 시 Presigned URL 처리:**

```
시간축 ──────────────────────────────────────────────────→

Phase 4 배포 시점
       │
       ▼
  ┌────┤────────────────────────────────────────────┐
  │ 기발급 R2 URL │          신규 NAS URL            │
  │ (최대 1시간   │     (NAS MinIO 가리킴)           │
  │  후 자연 만료)│                                   │
  └──────────────┴───────────────────────────────────┘

  • 기발급 R2 URL: R2가 아직 존재하므로 정상 동작 ✅
  • 신규 NAS URL: MinIO에서 서빙 ✅
  • 두 URL이 동시에 유효한 "공존 구간" = 최대 1시간
  • 공존 구간 중 데이터 정합성: Dual-Write로 양쪽 동일 보장
```

| 시나리오                      | 방식 A                                                           | 방식 B                                                                 |
| ----------------------------- | ---------------------------------------------------------------- | ---------------------------------------------------------------------- |
| 전환 직전 발급된 다운로드 URL | ❌ R2 URL이 NAS를 가리키지 않음 → **404 에러** (Railway 중지 후) | ✅ R2 URL은 R2가 살아있으므로 정상                                     |
| 전환 직전 발급된 업로드 URL   | ❌ R2에 업로드됨 → NAS에 누락                                    | ✅ Dual-Write 중이므로 양쪽에 존재                                     |
| 전환 직후 발급된 다운로드 URL | ✅ NAS에서 서빙                                                  | ✅ NAS MinIO에서 서빙                                                  |
| 멀티파트 업로드 진행 중 전환  | ❌ **업로드 실패** (R2 세션 무효)                                | ⚠️ Dual-Write Primary 전환 시 진행 중 멀티파트는 이전 Primary에서 완료 |

> **판정:** 방식 B는 R2를 유지하면서 전환하므로 Presigned URL 연속성이 **완전 보장**된다. 방식 A는 Railway 중지 시점에 기발급 URL이 무효화되어 **일부 요청 실패 불가피**.

#### 5.7.6 무중단 마이그레이션 종합 판정

| 평가 항목                | 방식 A (온프레미스)                     | 방식 B (터널 하이브리드)                            |
| ------------------------ | --------------------------------------- | --------------------------------------------------- |
| **마이그레이션 범위**    | 파일 + DB + API 전체                    | 파일 저장만                                         |
| **무중단 가능 여부**     | ❌ **불가** (최소 30분~2시간 계획 중단) | ✅ **가능** (서비스 중단 0분)                       |
| **마이그레이션 단계 수** | 3단계 (준비 → 동기화 → 빅뱅 전환)       | 5단계 (준비 → 복제 → Dual-Write → 읽기 전환 → 정리) |
| **각 단계 독립 롤백**    | ❌ Phase 3이 원자적 빅뱅 전환           | ✅ 각 Phase 독립 실행·검증·롤백                     |
| **롤백 소요 시간**       | 30분~수시간 (전환 후)                   | ~2~5분 (R2 보존 시)                                 |
| **롤백 난이도**          | ❌ DB 역동기화 + API 원복               | ✅ 환경변수 4개 원복                                |
| **데이터 손실 위험**     | ⚠️ 전환 후 역동기화 누락 가능           | ✅ R2 보존으로 데이터 손실 없음                     |
| **Presigned URL 연속성** | ❌ 기발급 URL 무효화                    | ✅ R2 URL + NAS URL 공존                            |
| **DB 정합성 위험**       | ❌ WAL 동기화 갭 발생 가능              | ✅ DB 이전 없음                                     |
| **필요 코드 변경**       | 8개 파일 전면 교체 (§5.4 참조)          | ~60줄 임시 코드 (안정화 후 제거)                    |
| **총 작업 기간**         | 3~5일 (점검 야간 작업 포함)             | ~1일 (서비스 중단 없음)                             |
| **운영 위험도**          | 🔴 높음                                 | 🟢 낮음                                             |

**최종 판정:**

1. **방식 A:** R2→NAS 무중단 마이그레이션을 **충족하지 못한다.** DB + API + 파일 저장을 동시에 전환하는 "빅뱅 마이그레이션"이 불가피하며, 최소 30분~2시간의 계획된 서비스 중단이 필요하다. 롤백도 전환 후에는 복잡하고 시간이 소요된다.

2. **방식 B:** R2→NAS 무중단 마이그레이션을 **충족한다.** Dual-Write → 읽기 전환 → R2 정리의 5단계 점진적 전환으로 서비스 중단 없이 이전이 가능하다. S3 호환 SDK(MinIO) 덕분에 코드 변경이 최소화되고, R2를 보존하면 어느 단계에서든 2~5분 내 즉시 롤백할 수 있다. 15GB 소량 데이터라 전체 복제도 1시간 내 완료된다.

> **권고:** 무중단 마이그레이션 요건이 있다면 **방식 B가 유일한 선택지**이다. 방식 A를 선택할 경우 야간 점검 공지 후 계획된 중단 시간을 확보해야 한다.

## 6. 종합 권고

### 6.1 최종 권고: 방식 B (터널 하이브리드) — NAS 전환 시 유일 권장 아키텍처

> **한 줄 요약:** NAS 전환을 결정한다면 **방식 B(터널 하이브리드)**가 유일하게 모든 필수 조건을 충족하는 아키텍처이다. 단, 현 시점에서는 R2 유지가 비용·안정성 면에서 최선이므로, **전환 시기는 데이터 주권 필요성 또는 데이터 규모 증가에 따라 판단**한다.

#### 최종 권고 근거 요약

| 필수 조건           | 방식 A (온프레미스)                                  | 방식 B (터널 하이브리드)                                       | 근거 섹션 |

| 무중단 마이그레이션 | ❌ **미충족** — 빅뱅 전환 필수, 30분~2시간 중단      | ✅ **충족** — Dual-Write 5단계, 서비스 중단 0분                | §5.7.6    |

**방식 B는 3개 필수 조건 중 2개 완전 충족, 1개 조건부 충족(R2 폴백 적용 시 충족)으로 유일한 권장 아키텍처이다.**

### 6.2 방식 A를 권장하지 않는 이유

방식 A(온프레미스 완전 이전)는 아래 구조적 한계로 인해 **yjlaser_website 웹하드 환경에서 채택할 수 없다:**

1. **SPOF(단일 장애점) 구조** — NAS 한 대에 API, DB, 파일 저장이 집중되어, 정전·ISP 장애·하드웨어 고장 중 어느 하나라도 발생하면 **전체 서비스가 중단**된다. 소규모 NAS에서는 HA(이중화) 클러스터 구성이 불가능하므로 구조적으로 해결할 수 없다 (§5.6.6).

2. **API 응답 속도 열화** — 현행 Vercel→Railway 경로(RTT ~50ms)가 Vercel→사무실 NAS 경로(RTT ~150ms)로 변경되어, 메타데이터 API 응답이 **3~5배 느려진다.** 파일 전송 속도는 100Mbps 회선 병목으로 양 방식 동등하지만, API 레이턴시 증가는 모든 페이지 로딩·폴더 탐색·검색 UX에 영향을 미친다 (§5.5.3).

3. **빅뱅 마이그레이션 강제** — DB + API + 파일 저장을 동시에 전환해야 하므로 점진적 이전이 불가능하다. 최소 30분~2시간의 계획된 중단이 필요하며, 전환 후 롤백도 DB 역동기화를 수반하여 복잡하고 위험하다 (§5.7.6).

4. **운영 부담 급증** — Docker + PostgreSQL + NAS + 네트워크 + 보안(SSL, DDoS, 방화벽)을 모두 자체 관리해야 하며, 이는 소규모 제조업체의 IT 역량 대비 과도한 요구사항이다 (§5.3.2 ②).

### 6.3 전환 필요성 판단

NAS 전환 자체의 필요성을 먼저 검토한다:

| 질문                        | 답변                              | 영향                 |
| --------------------------- | --------------------------------- | -------------------- |
| R2 비용이 부담인가?         | 아니오 (월 ~320원, $0.24)         | 비용 동기 없음       |
| 데이터 국내 보관 법적 요구? | DXF/PDF 도면 → 특별 규제 없음     | 법적 동기 약함       |
| 외부 서비스 장애 리스크?    | R2 SLA 99.9%, 실적 매우 안정      | 안정성 동기 약함     |
| 사내 LAN 고속 접근 필요?    | 사무실에서 대량 파일 작업 시 유리 | **유일한 실질 동기** |
| 데이터 자체 통제 의지?      | 경영 판단 사항                    | 정성적 동기          |

> **현 시점 결론:** 기술적·경제적으로 R2 유지가 최선이다. NAS 전환의 동기는 "데이터 자체 보유"라는 경영 판단 또는 향후 데이터 규모 증가(1TB+)에 한정된다.

### 6.4 시나리오별 실행 계획

#### 시나리오 1: R2 유지 (현행 유지) ⭐ **현 시점 최적**

**적용 조건:** 현재 전제 조건(15GB, 낮은 동시접속, 월 $0.24) 유지 시

- 변경 없음, 리스크 없음, 비용 최저
- 사내 LAN 속도 이점을 포기하는 대신 운영 안정성 확보
- **DS215j는 순수 내부 백업용으로 활용** (rclone으로 R2 → NAS 야간 백업)

#### 시나리오 2: 터널 하이브리드 (방식 B) ⭐ **NAS 전환 결정 시 유일 권고**

**적용 조건:** 데이터 자체 보유가 필수이거나, 데이터 규모가 1TB를 초과하여 R2 비용 절감이 필요할 때

- 파일 저장만 NAS로 이전, NestJS(Railway) + Next.js(Vercel) + PostgreSQL(Railway)은 현행 유지
- MinIO(S3 호환 오브젝트 스토리지)를 NAS Docker에서 구동 → **기존 S3 SDK 코드 호환, 환경변수 4개만 변경**
- Cloudflare Tunnel로 외부 접근 해결 (포트 포워딩 불필요, 보안 유지)
- R2를 폴백 스토리지로 보존 → NAS 장애 시 자동 전환으로 99.8%+ 가용률 확보
- **NAS 최소 사양:** DS923+ (4GB RAM, AMD Ryzen R1600, Docker 지원) — §4 참조
- **구현 기간:** 2~3주 (Dual-Write 점진적 전환, 서비스 중단 0분)
- **롤백:** 환경변수 4개 원복으로 2~5분 내 R2 복귀

#### 시나리오 3: 온프레미스 완전 이전 (방식 A) — ❌ 권장하지 않음

**부적합 사유:** 3개 필수 조건 모두 미충족 (§6.2 상세 참조)

- 모든 인프라를 사내에서 운영하려는 경우에만 고려 대상
- NAS가 SPOF가 되어 24/7 무중단 운영 불가
- API 레이턴시 3~5배 증가로 전체 UX 열화
- 빅뱅 마이그레이션 필수, 롤백 위험
- **이 방식을 선택해야 할 경우:** 반드시 UPS + 이중 ISP + 자체 모니터링 + 백업 전략을 갖추고, 야간 계획 중단을 공지한 후 전환 실행
