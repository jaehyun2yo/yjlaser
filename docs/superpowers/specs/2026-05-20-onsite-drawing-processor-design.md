# 현장도면가공프로그램 설계

## 목적

`현장도면가공프로그램`은 유진레이저목형 업무 흐름 중 현재 범용 CAD에서 수동으로
처리하는 도면 가공 작업을 대체하기 위한 현장가공 전용 CAD 프로그램이다. 주요
대상 단계는 `도면수정`, `DXF 가공`, `오시칼 작업`이며, 현장 작업자가 기존 CAD를
켜지 않고도 도면을 열고, 그리고, 수정하고, 유진레이저 작업 규칙에 맞게 자동
가공할 수 있어야 한다.

성능이 핵심 요구사항이므로 기술스택은 `C++20 + Qt 6 + CMake + vcpkg`를
기본으로 한다.

## 제품 방향

최종 지향점은 범용 CAD 전체를 복제하는 것이 아니라, 유진레이저 현장가공에 필요한
기능을 빠르고 안정적으로 제공하는 전용 CAD다.

| 단계 | 범위 | 설명 |
|---|---|---|
| 1차 제품 목표 | CAD 대체 MVP | 불러오기, 선택, 이동/삭제, 레이어/색 변경, 선/원/호/폴리라인/텍스트 신규 작성, 자동 오시/칼선/브릿지/합판 처리 |
| 확장 목표 | 고급 CAD 편집 | 치수선, 스냅, 정렬, 트림/연장 같은 CAD 편집 기능까지 확장 가능한 구조 |

현재 제품 목표는 1차 제품 목표다. 다만 첫 개발 작업은 전체 제품을 한 번에
완성하는 것이 아니라, 이 목표를 구현할 수 있는 프로젝트 골격과 핵심 편집 구조를
만드는 것이다. 아키텍처는 확장 목표의 기능을 나중에 추가할 수 있도록 명령 시스템,
편집 도구 시스템, 스냅/제약 확장 지점을 처음부터 둔다.

## 첫 개발 범위

첫 개발 범위는 실행 가능한 프로젝트 골격과 CAD 대체 MVP의 핵심 편집 구조를
만드는 것이다. 사용자 입장에서 완성된 CAD 기능을 모두 제공하는 단계가 아니라,
이후 기능을 빠르게 붙일 수 있는 안정적인 기반을 만드는 단계다.

- C++20/Qt 6 데스크톱 앱 프로젝트 생성
- CMake dev/prod preset 분리
- vcpkg 기반 의존성 관리
- 전역 로깅 시스템 도입
- 단위 테스트와 성능 벤치마크 실행 구조 도입
- DXF/AI 입출력 어댑터 인터페이스 정의
- 도면, 엔티티, 레이어, 스타일, 도면 경계 모델 정의
- CAD 캔버스와 도메인 코어를 분리하는 아키텍처 구성
- 선택, 이동, 삭제, 레이어/색 변경을 위한 command 기반 편집 구조
- 선, 원, 호, 폴리라인, 텍스트 신규 작성을 위한 tool 기반 입력 구조
- undo/redo를 위한 명령 이력 구조
- 향후 스냅, 치수선, 트림/연장 확장을 위한 service/port 경계

첫 개발 범위에서 실제 완성 기능으로 포함하지 않는 것은 다음과 같다.

- 완전한 DXF 파서 구현
- 모든 AI 파일 변형 지원
- 치수선, 정렬, 트림/연장 등 고급 CAD 편집 기능 완성
- 전체 범용 CAD 기능 복제
- 모든 엔티티 타입을 대상으로 한 브릿지 삽입 알고리즘 완성
- 네스팅 프로그램과의 직접 연동
- 배포 설치파일 제작

## 기술스택

| 영역 | 선택 |
|---|---|
| 언어 | C++20 |
| GUI | Qt 6 Widgets |
| 렌더링 | Qt 기반 CAD 캔버스, 이후 QOpenGLWidget 확장 가능 |
| 빌드 | CMake |
| 패키지 관리 | vcpkg manifest mode |
| 로깅 | spdlog |
| 테스트 | Catch2 |
| 벤치마크 | Google Benchmark |
| 기하 연산 후보 | Clipper2, Boost.Geometry |
| DXF 파서 후보 | libdxfrw 또는 자체 Adapter |
| AI 파서 후보 | PDF 호환 AI는 PDFium 계열, legacy EPS AI는 별도 변환 Adapter |

Qt 6를 기본으로 하되, Windows 7 호환성이 새 요구사항으로 추가되면 Qt 5로
내려가는 별도 의사결정이 필요하다.

## 아키텍처

```text
현장도면가공프로그램/
+-- CMakeLists.txt
+-- CMakePresets.json
+-- vcpkg.json
+-- README.md
+-- docs/
|   +-- architecture.md
|   +-- development.md
+-- src/
|   +-- app/              # Qt 앱 진입점, MainWindow
|   +-- canvas/           # CAD 캔버스, viewport, selection
|   +-- commands/         # undo/redo 가능한 편집 명령
|   +-- core/             # 도면/엔티티/레이어 도메인 모델
|   +-- geometry/         # bounds, intersections, bridge 후보 계산
|   +-- io/               # DXF/AI importer/exporter port와 adapter
|   +-- logging/          # spdlog 초기화, dev/prod log 정책
|   +-- services/         # 오시/칼선 분류, 합판 생성, 구아이 표기
|   +-- tools/            # 선택/이동/그리기 등 CAD 입력 도구
|   +-- config/           # 실행 모드, 설정 로드
+-- tests/
+-- benchmarks/
+-- cmake/
```

의존성 방향은 다음을 지킨다.

```text
app/canvas -> tools -> commands -> services -> core/geometry
io -------------------------------> core
logging/config은 모든 계층에서 사용할 수 있으나 도메인 규칙을 가지지 않는다.
```

## 핵심 도메인 모델

- `DrawingDocument`: 하나의 도면 문서. 엔티티 목록, 레이어 목록, 단위, 경계 정보를 가진다.
- `Entity`: LINE, ARC, CIRCLE, POLYLINE, SPLINE, TEXT 등 CAD 기본 객체의 공통 인터페이스.
- `Layer`: 레이어 이름, 색상, 출력 속성을 가진다.
- `EntityId`: 편집과 선택을 위해 안정적인 식별자를 가진다.
- `Bounds`: 전체 칼선 크기와 자동 합판 생성의 기준이 되는 경계 박스.
- `TextEntity`: 도면 내 텍스트 추출 대상.
- `Sheet`: 도면에 맞춰 자동 생성되는 합판 정보.
- `Command`: undo/redo 가능한 문서 변경 단위.
- `Tool`: 선택, 이동, 선 그리기, 원 그리기 등 사용자 입력 상태를 관리하는 단위.

도메인 모델은 Qt 타입에 직접 의존하지 않는다. UI와 렌더링 계층에서는 도메인
모델을 화면 좌표로 변환해서 사용한다.

## 기능 설계

### DXF, AI 읽기

입출력은 `DrawingImporter` 인터페이스로 시작한다.

- `DxfImporter`: DXF 파일을 `DrawingDocument`로 변환한다.
- `AiImporter`: AI 파일을 `DrawingDocument`로 변환한다.
- `ImporterRegistry`: 확장자와 파일 헤더를 기준으로 적절한 importer를 선택한다.

AI는 파일 변형이 많으므로 MVP 이후 두 단계로 나눈다.

1. PDF 호환으로 저장된 AI 파일 지원
2. legacy EPS 기반 AI 파일 지원

지원하지 못하는 AI 파일은 명확한 오류와 로그를 남기고, 사용자가 DXF/PDF 호환
AI로 다시 저장해야 함을 안내한다.

### 엔티티 수정

엔티티 수정은 명령 기반으로 설계한다. 모든 수정은 `Command`를 통해 문서에
적용되며, command는 실행 전/후 상태를 기록해 undo/redo가 가능해야 한다.

- 선택
- 이동
- 레이어 변경
- 색상 변경
- 삭제
- 속성 변경

### 엔티티 신규 작성

1차 구현에는 CAD 대체 MVP에 필요한 기본 작성 도구를 포함한다.

- 선
- 원
- 호
- 폴리라인
- 텍스트

각 작성 기능은 `Tool`로 구현한다. Tool은 마우스/키보드 입력 상태를 관리하고,
도형 생성이 확정되면 `CreateEntityCommand`를 만들어 문서에 반영한다. 이렇게 하면
나중에 스냅, 직교 모드, 길이 입력, 좌표 입력을 Tool에 확장할 수 있다.

### 고급 CAD 편집 확장

아래 기능은 1차 구현에서 완성하지 않지만, 구조적으로 추가 가능해야 한다.

- 치수선
- 객체 스냅
- 격자 스냅
- 직교/수평/수직 제약
- 정렬
- 트림
- 연장
- 오프셋

이를 위해 `SnapService`, `ConstraintService`, `ModifyTool`, `DimensionService`의
확장 지점을 둔다. 1차 구현에서는 인터페이스와 비어 있지 않은 최소 동작만 두고,
실제 고급 알고리즘은 후속 단계에서 구현한다.

### 오시선/칼선 자동 구분

`CutCreaseClassifier` 서비스를 둔다.

현재까지의 결정:

- 오시선/칼선 자동 구분은 1차 제품 목표에 포함한다.
- 오시선/칼선 구분과 지정 레이어/색 일괄 변경은 하나의 자동화 명령으로 묶는다.
- 이 기능은 오판 시 실제 가공 결과가 달라질 수 있으므로 기본 정책을 보수적으로 둔다.
- 확실히 판단한 엔티티만 자동 변경하고, 애매한 엔티티는 `미확정`으로 남겨 사용자가 검수한다.

중요한 보류 사항:

- 오시선과 칼선은 선 자체의 레이어/색/길이만 보고 안정적으로 구분하기 어렵다.
- 박스의 구조, 접힘 방식, 날개/뚜껑/바닥/접착부 같은 도무송 패키지 문맥을 알아야 판단할 수 있다.
- 따라서 오시선/칼선 자동 구분 규칙은 별도 브레인스토밍을 통해 박스 구조 지식을 먼저 정리한 뒤 확정한다.

후속 브레인스토밍에서 확인할 질문:

- 유진레이저에서 자주 작업하는 박스 유형은 무엇인가?
- 각 박스 유형에서 칼선, 오시선, 보조선은 어떤 위치와 형태로 나타나는가?
- 거래처 도면에서 오시선/칼선이 보통 어떤 레이어명, 색상, 선종류로 들어오는가?
- 같은 색/레이어라도 박스 구조상 다르게 해석해야 하는 경우가 있는가?
- 자동 판단이 틀리면 실제 제작에서 어떤 문제가 발생하는가?
- 사용자 검수 화면에서 미확정 엔티티를 어떤 방식으로 보여줘야 가장 빠르게 확인할 수 있는가?

초기 입력 후보:

- 엔티티 타입
- 레이어명
- 색상
- 선 길이/형상
- 닫힌 경로 여부
- 기존 CAD 관례 설정
- 박스 유형
- 박스 구조 내 위치
- 인접한 칼선/오시선과의 관계

초기 출력 후보:

- 칼선 레이어/색상
- 오시선 레이어/색상
- 미확정 엔티티 목록

문서상 확정하지 않는 것:

- 박스 구조별 오시/칼선 판별 알고리즘
- 레이어명/색상별 최종 판정표
- 미확정 엔티티 검수 UI

위 항목은 별도 브레인스토밍과 실제 샘플 도면 검토 후 확정한다.

### 브릿지 삽입

`BridgeInsertionService`가 일정 간격마다 칼선에 브릿지를 삽입한다.

현재까지의 결정:

- 브릿지 삽입은 1차 제품 목표의 개별 자동화 명령으로 제공한다.
- 사용자는 브릿지 설정 창에서 값을 조정한 뒤 선택한 범위에 적용할 수 있어야 한다.
- 기본 삽입 방식은 칼선 길이를 따라 `50mm마다 1개` 브릿지를 넣는 방식이다.
- 50mm 간격은 기본값일 뿐이며, 사용자가 설정에서 변경할 수 있어야 한다.
- 브릿지 폭은 설정 가능해야 하며, 기본값 후보는 `4.0mm`다.
- 끝 브릿지 폭은 별도 설정 가능해야 하며, 기본값 후보는 `2.0mm`다.
- 브릿지 개수, 폭, 끝 브릿지 폭, 간격 보정값은 직접 입력 가능해야 한다.
- 설정값은 자체 프로젝트 파일에 저장되어 같은 도면을 다시 열 때 유지되어야 한다.

브릿지 적용 범위:

- 객체 브릿지: 선택한 엔티티 또는 선택한 객체 묶음에만 적용한다.
- 그룹 브릿지: 사용자가 지정한 그룹/묶음 단위에 적용한다.
- 전체 브릿지: 현재 도면의 대상 칼선 전체에 적용한다.

설정 UI에 포함할 항목:

- 적용 범위: 그룹 브릿지 / 객체 브릿지 / 전체 브릿지
- 갯수
- 브릿지 폭
- 끝 브릿지 폭
- 브릿지 간격 또는 간격 보정
- 양쪽 / 한쪽 적용 여부
- 칼선/등선 자동 검색 여부
- 삭제하지 않을 항목 또는 보호 색상 설정
- 브릿지 설정 저장/불러오기

초기 기본값:

- 적용 범위: 객체 브릿지
- 브릿지 간격: 50mm마다 1개
- 브릿지 폭: 4.0mm
- 끝 브릿지 폭: 2.0mm
- 동작 정책: 보수적 적용, 애매한 엔티티는 미확정으로 남김

설계 원칙:

- 원본 엔티티는 직접 파괴하지 않는다.
- 선/폴리라인을 분할한 결과 엔티티를 새 문서 버전으로 생성한다.
- 브릿지 폭, 간격, 최소 선 길이, 끝 브릿지 폭은 설정값으로 둔다.
- 브릿지 삽입 전 preview를 제공하고, 적용 후 undo가 가능해야 한다.
- 브릿지 삽입 결과는 리포트에 남긴다. 리포트에는 대상 엔티티 수, 삽입 개수,
  제외된 엔티티 수, 제외 사유가 포함되어야 한다.
- 곡선 엔티티는 세그먼트 근사 후 처리하거나 전용 알고리즘으로 확장한다.

### 도면 정보 읽기

`DrawingAnalysisService`를 둔다.

현재까지의 결정:

- 도면 정보 읽기는 1차 제품 목표의 개별 자동화 명령으로 제공한다.
- 업체마다 도면에 들어있는 텍스트 형식과 위치가 다르므로, 먼저 도면 내 모든
  텍스트를 원문 그대로 추출한다.
- 원문 텍스트 추출 결과를 보존한 뒤, 그 위에서 주요 업무 필드 후보를 별도로
  구조화한다.
- 구조화 필드는 자동 추정 결과이며, 확신도가 낮거나 여러 후보가 있으면 사용자가
  확인할 수 있어야 한다.

분석 대상:

- 전체 칼선 기준 width/height 계산
- 도면 내 TEXT/MTEXT 전체 추출
- AI/PDF/EPS에서 변환된 텍스트 추출
- 레이어별 엔티티 수 집계
- 미확정 엔티티 목록 반환

원문 텍스트 추출 요구사항:

- 도면 내 모든 텍스트를 누락 없이 추출하는 것을 우선한다.
- 텍스트 값, 위치, 크기, 회전, 레이어, 색상, 원본 파일 포맷을 함께 보존한다.
- 읽기 실패 또는 깨진 텍스트는 실패 항목으로 리포트에 남긴다.
- 원문 텍스트 목록은 자체 프로젝트 파일에 저장한다.

구조화할 주요 필드:

- 업체명
- 제품명
- 인쇄지 종류
- 인쇄지 사이즈
- 정타/후타 여부

구조화 원칙:

- 자동 추출 필드는 `확정값`이 아니라 `후보값`으로 취급한다.
- 후보 추출 근거가 된 원문 텍스트를 함께 보여준다.
- 사용자가 직접 수정한 값은 자체 프로젝트 파일에 저장한다.
- 업체별 표기 패턴은 추후 규칙/프리셋으로 확장 가능해야 한다.

### 도면에 맞는 합판 자동 제작

`SheetGenerationService`가 업체별 자동판/동판 규격과 도면 정보를 바탕으로 합판
도형을 생성한다. 구아이 10mm 선 하단 기입은 독립 자동화로 실행하지 않고, 합판
자동 생성 흐름에 포함한다.

현재까지의 결정:

- 합판 자동 생성은 1차 제품 목표의 개별 자동화 명령으로 제공한다.
- 구아이 10mm 선 하단 자동 기입은 합판 생성 명령의 하위 단계로 통합한다.
- 다만 단순히 도면 외곽에 여백을 더해 사각형을 만드는 기능만으로는 부족하다.
- 업체별 자동판, 동판 규격이 있으며, 이 규격에 따라 생성해야 할 합판 기준이 달라진다.
- 따라서 합판 자동 생성 규칙은 별도 브레인스토밍을 통해 업체별 규격과 판 종류를 먼저 정리한 뒤 확정한다.

중요한 보류 사항:

- 업체별 자동판/동판 규격 목록
- 자동판과 동판을 구분하는 기준
- 도면 정보 또는 거래처명에서 업체 규격을 자동 선택하는 방식
- 규격이 여러 개일 때 기본 선택 기준
- 규격과 맞지 않는 도면을 열었을 때 경고/수동 선택 UX
- 규격 데이터 저장/수정/버전 관리 방식
- 구아이 10mm 선을 합판 하단 어디에, 어떤 텍스트/표식으로 기입할지
- 업체별 자동판/동판 규격에 따라 구아이 위치가 달라지는지

후속 브레인스토밍에서 확인할 질문:

- 업체별로 어떤 자동판/동판 규격이 있는가?
- 규격은 폭/높이만 중요한가, 여백/기준점/방향/구아이 위치까지 포함하는가?
- 도면 파일명, 도면 텍스트, 거래처명 중 무엇으로 업체를 판별할 수 있는가?
- 같은 업체가 여러 규격을 쓰는 경우 사용자는 어떻게 선택하는가?
- 잘못된 규격으로 합판을 만들면 실제 제작에서 어떤 문제가 생기는가?
- 업체별 규격은 누가, 어떤 화면에서 수정해야 하는가?
- 구아이 10mm 선은 합판 기준 하단에 항상 들어가는가, 업체/판 종류별 예외가 있는가?
- 구아이 표기는 도면 경계 기준인지 합판 경계 기준인지?

초기 입력 후보:

- 칼선 기준 bounds
- 도면 내 텍스트
- 파일명 또는 거래처명
- 업체 규격 프리셋
- 자동판/동판 종류
- 구아이 10mm 선 감지 결과

초기 출력 후보:

- 선택된 업체 규격
- 생성된 합판 도형
- 구아이 하단 표기
- 규격 불일치 경고
- 수동 확인 필요 목록

기본 설정 후보:

- 기본 여백
- 최소 합판 크기
- 폭/높이 반올림 단위
- 합판 레이어/색상

문서상 확정하지 않는 것:

- 업체별 자동판/동판 규격표
- 자동판/동판 자동 선택 알고리즘
- 규격 관리 UI
- 구아이 하단 기입 위치/표기 규칙

위 항목은 별도 브레인스토밍과 실제 업체 규격 자료 검토 후 확정한다.

### 불필요한 레이어/보조선 정리

`CleanupService`가 도면에서 불필요한 레이어, 보조선, 참고 요소를 정리한다.

현재까지의 결정:

- 불필요한 레이어/보조선 정리는 1차 제품 목표의 자동화 후보에 포함한다.
- 다만 정리 규칙은 지금 확정하지 않는다.
- 오시/칼선, 브릿지, 도면 정보 분석, 합판 생성 자동화가 어느 정도 동작한 뒤 실제
  결과물을 보고 사용자가 필요한 정리 규칙을 추가한다.
- 처음부터 자동 삭제 규칙을 강하게 넣으면 필요한 도면 요소를 잘못 지울 위험이 있다.

후속 브레인스토밍에서 확인할 질문:

- 자동화 결과물에서 실제로 자주 남는 불필요 요소는 무엇인가?
- 어떤 레이어/색상/엔티티 타입은 항상 삭제해도 되는가?
- 어떤 요소는 숨김 처리만 하고 삭제하면 안 되는가?
- 거래처 로고, 도면틀, 치수선, 주석, DEFPOINTS 계열 레이어는 어떻게 다룰 것인가?
- 삭제/정리 전 preview와 undo가 필요한가?
- 정리 규칙을 업체별로 다르게 가져갈 필요가 있는가?

초기 정책 후보:

- 기본은 비파괴 preview 후 사용자 승인
- 정리 규칙은 사용자 설정/프리셋으로 저장
- 삭제된 엔티티 수와 삭제 사유를 리포트에 기록
- 자동 시퀀스에 포함하더라도 첫 버전에서는 경고/확인 단계를 둔다

문서상 확정하지 않는 것:

- 삭제할 레이어명/색상/엔티티 타입 목록
- 보조선 판별 알고리즘
- 업체별 cleanup preset

위 항목은 자동화 기능이 어느 정도 구현된 뒤 실제 결과물을 보고 확정한다.

## 개발 모드와 프로덕션 모드

`CMakePresets.json`에서 빌드 모드를 분리한다.

- `dev`: Debug 빌드, 상세 로그, assert 활성화, 테스트/벤치마크 빌드
- `prod`: Release 빌드, 파일 로그 제한, assert 비활성화, 최적화 옵션 활성화

프로그램 실행 시에도 `APP_ENV=development|production` 또는 설정 파일을 통해
런타임 모드를 구분한다.

## 로깅

모든 주요 기능은 spdlog 기반으로 로깅한다.

- 앱 시작/종료
- 파일 import/export
- 엔티티 분류 결과
- 브릿지 삽입 결과
- 도면 분석 결과
- 예외/오류
- 성능 측정 구간

개발 모드는 콘솔 + 파일 로그를 사용하고, 프로덕션 모드는 회전 파일 로그를
사용한다.

## 성능 테스트

벤치마크 대상은 다음으로 시작한다.

- DXF import 시간
- 엔티티 bounds 계산
- 오시/칼선 분류
- 브릿지 삽입
- 캔버스 렌더링 데이터 준비
- command 실행/undo/redo 비용
- 기본 작성 도구의 preview geometry 생성 비용

벤치마크 데이터는 `benchmarks/fixtures/`에 작은 샘플과 대형 샘플을 분리한다.
실제 거래처 도면은 개인정보와 영업정보가 포함될 수 있으므로 fixture로 커밋하지
않는다.

## 검증 전략

- `cmake --preset dev`
- `cmake --build --preset dev`
- `ctest --preset dev`
- `현장도면가공_benchmarks` 실행

기능별로 단위 테스트를 먼저 추가하고, CAD 파일 입출력은 fixture 기반 통합 테스트를
둔다.

## 리스크와 결정사항

- AI 파일은 포맷 변형이 많아 전체 지원을 한 번에 약속하지 않는다.
- CAD 캔버스는 단순 Qt 렌더링으로 시작하되, 대형 도면에서 버벅이면 QOpenGLWidget
  기반 렌더러로 분리한다.
- DXF 파서는 외부 라이브러리를 우선 검토하되, 업무에 필요한 엔티티만 안정적으로
  처리하는 Adapter 경계를 둔다.
- 네스팅 프로그램과 기능이 겹치지 않도록 이 프로젝트는 현장 가공용 CAD 편집과
  자동 가공 명령에 집중한다.
- 범용 CAD 전체 기능을 한 번에 만들지 않는다. 1차 구현은 기존 CAD를 켜지 않고
  현장가공에 필요한 도면 작성/수정/자동처리를 수행하는 데 집중한다.
