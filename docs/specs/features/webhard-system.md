# Webhard System — Reference Spec

## Status: COMPLETE (reference document)

## PRD Reference: WH-001 ~ WH-009

## Architecture

- Frontend: `src/app/webhard/` — Container/Presentational pattern with React Context + hooks
- Backend: `webhard-api/src/` — NestJS modules (Files, Folders, Trash, Search, Storage)
- Storage: Google Shared Drive for new webhard files/folders through the NestJS storage provider boundary. Cloudflare R2 remains as a legacy/source compatibility provider for old upload producers and non-webhard assets.
- DB: Prisma tables (webhard_files, webhard_folders)

## Key Patterns

### File Upload Flow

```
Client → POST upload session request to NestJS → PUT directly to Google Drive resumable upload URL → POST metadata to NestJS
```

Folder upload follows the same direct-to-Drive rule for new webhard files. The Next.js folder upload Server Action may create folder structure and finalize metadata, but it must not read `File.arrayBuffer()` or materialize full browser files into `Buffer`. The browser groups files by authorized `folderId`, runs folder groups with bounded concurrency, and uses `uploadFilesBatch` for upload session, Drive PUT, and metadata completion. Batch upload session and batch-complete requests must receive the same abort signal as the Drive PUT work.

The Next.js `POST /api/webhard/upload/batch-complete` proxy treats metadata confirmation as failed when the NestJS response is non-2xx, when the response body has `success: false`, or when `failed > 0`. These cases return a failed JSON body with a non-2xx status so browser upload flows and E2E tests cannot report a partial metadata failure as a successful upload.

### Auth

- Company users: NestJS 세션 쿠키 (bcrypt + httpOnly cookie) → company_id isolation
- Admin: Cookie session → full access
- External programs: X-API-Key header → timing-safe compare

### Frontend Structure

```
src/app/webhard/
├── components/
│   ├── containers/     # Smart components (data fetching)
│   ├── presentational/ # Dumb components (display only)
│   └── context/        # WebhardContext (selection, navigation state)
├── hooks/              # Custom hooks (useWebhardFiles, etc.)
└── page.tsx
```

### Backend Modules

```
webhard-api/src/
├── files/        # CRUD, presigned URLs, metadata
├── folders/      # CRUD, tree operations, move
├── trash/        # Soft delete, restore, permanent delete
├── search/       # Full-text search across files/folders
├── storage/      # R2 legacy, Google Drive provider, quota, consistency diagnostics
├── share-links/  # Shared download links (token-based)
├── erp/          # ERP integration (machines, tasks, orders)
└── integration/  # External sync, webhook handling
```

## Conventions

- New upload operations use Google Drive resumable upload URLs. Google Drive downloads stream through NestJS/Next proxy because Drive media downloads require server credentials; R2 legacy files may still use presigned URLs.
- Batch upload session generation should share folder access checks, external-webhard routing results, and Drive target readiness within the request. For Drive-backed batches, pre-generate Drive file ids with `generateDriveIds(count)` and pass each id into resumable session creation so the provider does not issue one id-generation request per file.
- Drive batch upload confirmation verifies Drive metadata with bounded concurrency and caches per-folder Drive readiness within the request. A failed metadata/readiness check stays fail-closed for that file, but independent files in the same batch should not wait on sequential Drive confirmation.
- Google Drive upload proxy may return a signed `driveUploadProof` after a successful Drive PUT. Confirm paths may use this proof to skip the confirm-time Drive metadata GET only when the proof signature, storage file id, expiry, and expected parent digest all match. Missing or invalid proof falls back to the existing fail-closed Drive API verification path; proof payloads must not expose raw parent ids, secrets, presigned URLs, or Drive upload URLs in logs.
- Browser upload completion must not block the visible upload state on follow-up badge/storage/list refetches. After metadata confirmation succeeds, cache refreshes should run through targeted invalidation/background refetch and log failures without exposing presigned URLs, Drive ids, or raw paths.
- Upload confirm, batch confirm, file move, and file delete responses may move non-authoritative follow-up work to background execution: notifications, activity logs, cache invalidation, and auto-contact hooks. Google Drive object creation/move/trash, DB metadata writes, ownership checks, and storage repair recording remain authoritative and must complete before returning success.
- Folder upload completion follows the same UX contract: visible completion must not wait for follow-up file/folder/badge refetches. Cache invalidation should run in the background and log sanitized failures through the project logger.
- Share links for Google Drive files are token-based metadata records. The browser-facing Next route must generate a token and absolute `expiresAt`, validate file ownership for company users, forward the browser session when creating/listing share links, and ignore untrusted company id values for company users. NestJS share creation must resolve caller-supplied `webhardFileId` from DB and enforce the session company against the resolved file owner. Public token downloads must use the dedicated `POST /share-links/download/stream` API-key endpoint, and `maxDownloads` must be enforced by conditional atomic increment. Generic file download endpoints must not allow API key-only direct file access.
- ZIP download is browser-session scoped. It may include Google Drive files by streaming them from the storage provider into `archiver`, but API key integration principal is not enabled for ZIP downloads.
- `storageProvider=GOOGLE_DRIVE` rows must be backed by concrete Drive objects. Folders require `driveFolderId`; files require `driveFileId`. Rows that violate this are data drift, not displayable webhard content.
- Webhard list/tree/search APIs keep PostgreSQL metadata as the source of truth, but defensively exclude invalid Google Drive rows (`GOOGLE_DRIVE` with missing Drive id) from normal responses. Creation/confirm/copy paths must fail closed or explicitly store `R2`; they must not create DB-only Google Drive metadata.
- Webhard list/tree/search APIs are the DB-only fast path. They must not call Google Drive API, and must rely on indexed metadata, pagination, virtualized frontend lists, scoped aggregates, and realtime invalidation. Drive API calls are reserved for mutations, downloads, admin diagnostics, and background reconciliation.
- Company isolation via `company_id` on all queries
- Soft delete to trash first, permanent delete after 30 days
- Company root folders (`companyId != null`, `parentId == null`) are lifecycle-owned by the company record. Admin webhard delete must reject direct deletion with `COMPANY_ROOT_FOLDER_DELETE_BLOCKED`. The web UI must show a common modal containing the matched folder name and company name, offer navigation to `/admin/companies/:id`, and, for mixed selections, allow deleting the remaining non-matched items while excluding the matched company root folder.
- Company deletion is the authoritative path for deleting a matched company root folder. It sets `companies.status='deleted'`, disables webhard access, stores previous status/access for restore, and marks all matched root descendants with `deletedBy='company:{id}'`.
- During company deletion or restoration, a missing Google Drive folder id or a Google Drive 404 is treated as storage drift: record a sanitized storage repair event and continue the DB company delete/restore and webhard trash/restore transaction. Google Drive permission/configuration failures still fail the operation.
- Company restore is allowed only within 30 days and restores only webhard folders/files carrying that company deletion marker so ordinary trash items are not resurrected.
- Socket.IO for real-time file status updates
- Company users never receive Google Drive permissions or durable Google Drive folder links. Drive ids are server-owned metadata used for storage operations.
- Company approval/activation provisions a Drive root folder and configured template folders. If Drive and DB drift, the authoritative workflow records a sanitized `sync_logs.metadata.auditKind='storage_repair'` event.
- Admin diagnostics for storage drift are exposed through `GET /api/v1/storage/webhard-consistency`. The endpoint reports missing Drive ids, duplicate active company roots, optional sampled Drive API 404 checks, `quotaBackoffCount`, `lastCheckedAt`, and recent sanitized `storage_repair` events. Offline cleanup uses `webhard-api/scripts/audit-google-drive-webhard-consistency.ts` in dry-run mode first, then `--apply` when deleting invalid development fixture metadata is intended.
- Google Drive change notifications use `POST /api/v1/storage/drive-change-webhook`. The endpoint is separated from browser/API-key webhard routes, validates `GOOGLE_DRIVE_WEBHOOK_TOKEN` and optional `GOOGLE_DRIVE_WEBHOOK_CHANNEL_ID`, and only enqueues reconciliation work plus a sanitized repair event. It must not perform user-visible list verification inline.
- Background reconciliation is disabled unless `GOOGLE_DRIVE_RECONCILIATION_ENABLED=true`. When enabled, it checks a bounded number of recent Drive-backed folders/files per interval, records Drive 404 as `reason='drive_api_404'`, Drive 403/429 as `reason='drive_quota_or_backoff'`, and other failures as `reason='drive_api_error'`.
- Google Drive webhard release QA is covered by `e2e/google-drive-webhard-user-qa.spec.ts`. The Chromium scenario provisions two companies, verifies admin/company isolation, Drive-backed folder/file operations, share link, single-file ZIP, trash/restore, company delete/restore, and final storage consistency diagnostics. Manual QA remains for login form, ordinary folder restore, sorting UI, worker download, multi-file/folder ZIP, and deliberate Drive 404 injection.
- Pipeline observability: routing failures and AutoContact skip reasons are recorded as sanitized `sync_logs.metadata.auditKind='webhard_pipeline'` events and surfaced via `GET /integration/sync-logs/pipeline-backlog` (see `webhard-pipeline-observability.md`)
- AutoContact company resolution: when an uploaded file's folder has no `companyId`, folder traversal skips structural folders but may treat a top-level folder as a company root if its name resolves through `Company.companyName` or an approved `CompanyFolderAlias`. Structural roots such as `외부웹하드`, upload/download-only folders, request folders, and completion folders remain excluded.
- AutoContact realtime: every newly created AutoContact must emit `contact:created` through `ContactsGateway` to admin and worker rooms with id, numbers, company, stage, source, inquiry type, drawing filename, folder id, and timestamps. Duplicate file re-upload updates remain update-only and do not create a new-contact alert.
- AutoContact classify failed notification dedupe: when an uploaded file cannot be classified, admin `webhard_classify_failed` notifications are deduped by identical `metadata.folderPath` for 1 hour. The normal `new_contact` notification is still created for the new contact; only the classify-failed operational alert is suppressed.
- Batch AutoContact performance: `batchTriggerAutoContact` reuses the caller-provided folder map, batch-loads company names, caches folder path/company resolution, and starts distinct company+filename groups with bounded concurrency. Items with the same company+filename stay sequential to preserve duplicate-contact protection.
- Folder loading: default `/folders` queries are root/children lazy loading only. Breadcrumbs use `/folders/:id/ancestors`; full-tree reads must explicitly call `/folders/tree` or `/folders?includeAll=true` for compatibility/cache use cases.
- Folder loading must not automatically prefetch file lists for every visible child folder after a children response. File list prefetch is allowed only for explicit user intent such as hover or actual folder navigation, so a folder transition does not fan out into multiple unrelated `/files` requests.
- Folder tree expansion must share React Query cache keys with the rest of the webhard screen. Use `queryKeys.webhard.folders.children(parentId)` and `queryKeys.webhard.folders.ancestors(folderId)` instead of direct uncached `fetch()` calls, and do not cache failed responses as empty children or empty ancestors.
- Folder path updates: folder rename/move must update the root row and descendant materialized paths in the same transaction. Descendants are updated by set-based prefix replacement with a slash-boundary guard so sibling branches that merely share the same prefix are not modified. Folder rename/move never rewrites `WebhardFile.path`/R2 object keys.
- Folder move should reuse the already authorized target parent row for path materialization and Drive parent metadata. R2/DB-only moves must not perform Google Drive parent lookups; Drive-backed moves still verify and move the concrete Drive folder before committing DB metadata.
- Drive-backed batch file move should pass known source folder Drive ids to the storage provider. If the DB source parent has a `driveFolderId`, pass it as `fromParentStorageFolderId` so Google Drive move does not need a per-file metadata read to discover the old parent.
- Drive-backed batch file move/delete should use provider batch operations when all required Drive parent metadata is known. Batch results are itemized; any Drive item failure keeps the DB mutation fail-closed and records sanitized elapsed/success/failure counts for performance diagnosis.
- File delete/move UI paths should optimistically remove affected files from the active folder cache and rollback that cache on API failure. Follow-up badge/storage/list refetches should use targeted invalidation instead of blocking the first visible UI response.
- Folder batch delete should collect descendants with the selected folder `path` prefix and the `path` index instead of reading the entire folder table. Only `path=null` legacy selected folders should use the parent-map fallback.
- Undownloaded badges: `GET /files/badge-counts` is the single source of truth for total and propagated folder counts. Direct file counts stay scoped to the effective `companyId`; folder parent propagation uses that company folder tree plus legacy `companyId=null` bridge folders so counts inside visible admin-created folders still reach the visible company top-level folder. Frontend cache keys must include `companyId` plus `includeFolderCounts` so admin/company views never share badge data. Upload, download confirmation, delete, file move, folder move, folder realtime updates, and manual refresh invalidate the `queryKeys.webhard.badgeCounts()` prefix and let active scoped badge queries refetch the authoritative self/parent/root counts.
- Storage usage: the webhard sidebar uses `queryKeys.webhard.storageAll()` as the invalidation prefix. File upload, active-file delete/restore, and file realtime events must invalidate this prefix so the active storage bar refetches immediately. The display also refetches on window focus and uses a 30s polling fallback. Admin all-company active usage is the sum of every `company_storage.used_bytes` row, not only the `companyId=0` row; trash usage is aggregated from `webhard_files.size` where `deletedAt IS NOT NULL`, and the displayed total is `active + trash` because trashed files still occupy storage. If the trigger table is unavailable, aggregate active `webhard_files.size` instead.
- Backend use-case services: keep controller-facing `FoldersService`/`FilesService` methods stable, but route folder path materialization through `FolderPathService` and undownloaded badge aggregation through `BadgeCountsService`. New backend behavior in these areas should add focused use-case tests before changing the larger service tests.
- Frontend query/cache conventions: production webhard code must use `queryKeys.webhard.*` instead of raw `webhard` React Query key literals. The local webhard screen keeps `VirtualizedFileList` as the active virtual list; `src/lib/webhard-ui/components/VirtualFileList.tsx` remains a library export, not the screen implementation.
- Normal folder files: `/files` responses are consumed as paginated `{ files, pagination }` data. The webhard screen keeps one cache entry per folder/search/sort scope and appends `page + 1` into that cache when `hasMore` is true. Folders with more than 50 visible files must render through `VirtualizedFileList` and must not depend on eager full-folder reads.
- Recovery conventions: webhard UI recovery should reset the error boundary and invalidate/refetch relevant queries. Do not use `window.location.reload()` for state repair; socket reconnect failures should retry the namespace connection directly.
- `WebhardMain` split conventions: server data reads live in dedicated hooks (`useWebhardFilesQuery`, `useWebhardFoldersQuery`) and upload-after-link prompt state lives in `useWebhardUploadPrompt`. Query key, drag payload, context menu selected-count, and virtual list threshold contracts are centralized in `webhardMainContracts` and covered by tests before further `WebhardMain` refactors.
- Webhard main render gate: folder children and file list must be rendered together. The main list keeps the loading skeleton until the active file query and folder children query for the selected folder are both ready.
- Realtime move events: file move and batch move must notify both source folder subscribers and target folder subscribers. A moved file must disappear from the source page and appear in the target page without a full browser reload.
- Worker drawing revisions: additional drawing upload creates a `DrawingRevision`, registers each uploaded file as `WebhardFile` in the contact inquiry folder, emits `file:created` to that folder, and invalidates worker timeline/contact/process-board/webhard caches. The worker timeline server action must bypass ISR/revalidate caching so refetches see the new revision immediately.
- Worker new-contact notification: Worker dashboard listens to `/contacts` `contact:created`; it shows an in-session notification list before/refetch in addition to invalidating field/office/unclassified process board queries. Notification click routes by source/type/stage and scrolls to `worker-contact-{contactId}`.
- Worker inquiry-card filenames: worker cards display `companyName - fileName` as a presentation-only transform and do not rewrite `WebhardFile.name` or R2 keys. The download button uses the `latest-drawing-url` proxy and treats the last uploaded `DrawingRevision` as the latest drawing, independent of the current process stage. Browser download filenames use `inquiryNumber - companyName - fileName` when an inquiry number exists, falling back to work number only when needed. Existing `[O]`/`[F]`, full download prefixes, and short `O-001`/`F-001` prefixes are stripped before the final name is assembled.
- Website contact uploads: public `/contact` submissions create the Contact first, then upload attachment, drawing, and reference photo files to the Google Drive inquiry folder through the NestJS storage boundary. `ensureInquiryFolder(contactId)` is called before upload so the company inquiry folder exists even when the initial Contact create path did not create it. Each uploaded file is registered as a `WebhardFile` with `storageProvider=GOOGLE_DRIVE`.
- Inquiry folder names: contact inquiry folders use number-only names under `{company}/문의/`. O-only stores `{O}`, F-only stores `{F}`, and O+F stores `{O}_{F}` because `/` is not valid in folder names. Worker/admin webhard UI displays the same contract as `O / F`, `O /`, or `/ F`.
- Worker current webhard location: worker contact list responses use the latest DrawingRevision file's current `WebhardFile.folderId` and folder path, then the `contactId + folderKind='inquiry'` folder, before falling back to `Contact.webhardFolderId`. Context-menu "웹하드에서 열기" and the card storage path must therefore point to the current inquiry folder after relocate.
- Company dashboard contact cards: the `웹하드` action appears before `메모` and opens `buildWebhardUrl(webhard_folder_id, webhard_file_id)`. `/contacts/by-company` must resolve `webhard_folder_id` from the latest DrawingRevision file's current folder, then the `contactId + folderKind='inquiry'` folder, before falling back to `Contact.webhardFolderId`, so the action opens the inquiry folder rather than the company root. If `webhard_folder_id` is absent, the action stays disabled instead of guessing a folder. Company-facing inquiry and booking card titles display the package/inquiry title without the leading company-name prefix; this is a presentation-only transform and does not rewrite `Contact.inquiryTitle`. Company dashboard card actions, including `웹하드`, `메모`, `예약변경`, and `예약취소`, share the `CardActionButton` visual contract so the action group stays visually consistent.
- Completed inquiry folders: delivery completion moves the inquiry folder under `{company}/문의/완료/`. The backend creates `문의/완료/` when missing and normalizes legacy inquiry folders that still sit under root-level `{company}/완료/`.
- Delivery proof files: when delivery completion includes a proof photo, the backend uploads the image to the contact Google Drive inquiry folder and creates a `WebhardFile` named `납품완료_YYYYMMDD_HHmmss.ext` with `storageProvider=GOOGLE_DRIVE`. The Contact `deliveryProofImage` value stores the Drive storage reference. Admin/company/worker UI must render through `DeliveryProofImage` or `/api/contacts/:id/delivery-proof`, which returns a same-origin stream URL for Drive files. The folder receives a `file:created` realtime event.
- Server-side NestJS forwarding: server actions and Next API proxies must sanitize forwarded `Cookie` headers before calling NestJS. Cookie names/values that contain Korean or other non-ASCII characters are percent-encoded so Node `fetch` can construct ByteString-compatible request headers; CSRF token extraction continues to read from the raw cookie string before sanitizing.
- Company-facing contact/bookings proxies: Next.js routes that call NestJS with a server API key must validate the browser session actor before the server-to-server call. Company sessions may only access records whose `company_name`/`companyName` matches the session company resolved from the signed company session; booking mutations with `contactId` must also verify the linked contact belongs to the same company as the booking. Mutation routes that represent admin workflows stay admin-only.
- Company-facing timeline proxy: `/api/contacts/:id/timeline` must not fetch NestJS timeline data with a server API key for company sessions. It must forward the browser session cookie so NestJS applies the company timeline filter that removes private drawing revisions and masks internal notes.
- Revision request upload order: `/api/contacts/:id/revision-request` must verify contact ownership before upload work. Authorized file uploads go to the Google Drive inquiry folder through `POST /api/v1/contacts/:id/revision-request-file/drive`; unauthorized company requests must not create orphan Drive files.
- Drawing revision download proxy: `/api/drawing-revisions/:revisionId/download` must resolve revision access metadata before issuing the presigned URL. Admin sessions may download any revision, company sessions may download only public revisions whose contact company matches the signed session company, and ERP worker sessions remain allowed for worker timeline downloads. `/api/drawing-revisions/:revisionId/visibility` is admin-only.
- Worker-facing webhard/drawing proxies: `/api/worker/files`, `/api/worker/files/:id/download`, `/api/worker/drawing-revisions`, and `/api/worker/drawing-revisions/upload-urls` must call `getErpWorkerSession()` before any backend call. A forged or unverifiable `erp-session` returns 401 and must not fetch presigned URLs, file lists, drawing history, or upload URLs. Worker file read and drawing revision calls forward the verified worker session cookie to NestJS instead of using `X-API-Key`; drawing mutation calls also forward CSRF token. These proxies must forward only `erp-session` and `csrf-token`, never the raw browser Cookie header, because stale admin/company cookies can make NestJS select the wrong actor and reject the request with `Verified worker session required`. NestJS only enables worker session auth on explicit read/mutation routes.
- Company webhard access gate: `/webhard` must fail closed for company users if `GET /folders/company-info/:companyId` fails or returns null. Do not render `WebhardMain` until `webhardAccess` and root-folder state are known.
- Backup proxy auth: `/api/admin/backup/*` verifies the Next.js admin session and forwards the sanitized admin cookie to NestJS. It must preserve upstream `Set-Cookie` headers so NestJS can bootstrap CSRF cookies through the proxy. It must not rely on `MIGRATION_API_KEY`; API key calls to NestJS backup endpoints require explicit `backup:*` scopes per the endpoint spec.
- Browser download/preview proxy: file download proxy responses must sanitize non-ByteString response headers such as Korean `Content-Disposition`. File hover must not open previews. File double-click and "다운로드" menu actions always start download. DXF preview uses the Next route `/api/webhard/preview-dxf?fileId=` through the separate "미리보기" context-menu action to resolve the NestJS download URL server-side and return the DXF text without exposing the presigned URL, and the preview modal exposes a download action next to reset/original-size controls.
- Role-based UI actions: company users cannot create folders and cannot delete or move folders. Company users should not see folder context-menu actions, folder drag/drop handles, or file delete buttons; any remaining delete attempt path must be blocked before the API call and show "관리자에게 삭제 요청해주세요". Company file actions remain download, rename, move, share-link creation, and preview when supported.
- Company folder visibility: when an admin creates a child folder under a company-owned parent, the child inherits the parent `companyId` if the request omits `companyId`. Company folder children queries also keep legacy `companyId=null` child folders visible under the selected company parent while retaining the external-webhard visibility filter.
- Main-area folder interactions: file and folder rows use the same activation model. Single click selects the item; if another item is already selected, a plain single click adds the clicked file/folder to the current selection instead of replacing it. File double-click starts download, and folder double-click enters that folder. Admin folder items in the main file area support folder drag/drop move and the same folder context menu actions as the tree where applicable, including rename and delete. Folder drag payloads use the dedicated `application/x-folder-id` MIME entry. Company users must not be able to drag folders.
- Main list metadata/sorting: folder rows display the same upload date/uploader columns and file-meta typography as files. Folder upload date is the latest `created_at` among descendant files; if no descendant file exists, the UI displays the folder `created_at`. Folder uploader is the latest descendant file uploader display name and falls back to company name or `-`. File name, upload date, and uploader sorting apply to both folders and files in the main list.
- Badge visual contract: one-digit folder/new badges render in a fixed circular box with zero padding so the number stays visually centered.
- Session expiry UX: if an already-open webhard page receives 401 or 419 from file/folder list queries after idle session expiry, the webhard UI must redirect to `/login` instead of rendering the authenticated empty-folder copy. Auth errors are represented as `WebhardApiError` and should not be hidden by empty-array fallback or repeated React Query retries.
- Browser-facing webhard API rate limiting: `checkWebhardRateLimit()` is disabled when `NODE_ENV !== 'production'` so local/test uploads and badge refreshes are not blocked by development traffic. Production keeps the configured Upstash/in-memory limiter, and Upstash `reset` values are treated as absolute Unix timestamps in milliseconds.
