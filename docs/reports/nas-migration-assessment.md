## 2. 기존 NAS(DS215j) 적합성 판단

### 하드웨어 스펙

| 항목         | DS215j 스펙                                    | 웹하드 서버 최소 요구              |
| ------------ | ---------------------------------------------- | ---------------------------------- |
| **CPU**      | MARVELL Armada 375, 800MHz, 2코어 (ARM 32-bit) | x86_64 또는 ARM64, 1.4GHz+ 4코어   |
| **RAM**      | 512MB (증설 불가)                              | 최소 4GB (Docker + NestJS + MinIO) |
| **아키텍처** | ARMv7 (32-bit)                                 | x86_64 필수 (Docker 이미지 호환성) |
| **DSM**      | 6.2.4 (EOL 임박)                               | DSM 7.2+ (Container Manager 지원)  |
| **Docker**   | 미지원                                         | 필수 (NestJS 컨테이너 구동)        |
| **디스크**   | 2-bay                                          | 2-bay 이상 (RAID 1 미러링 권장)    |

### 판정: 불가

DS215j로 웹하드 파일 서버를 운영할 수 없는 이유:

1. **Docker 미지원** — NestJS 백엔드를 컨테이너로 구동할 수 없음. Synology Docker(Container Manager)는 x86_64 CPU + DSM 7.x 이상에서만 지원됨. DS215j의 ARM 32-bit CPU는 근본적으로 호환 불가.

2. **RAM 부족** — 512MB RAM으로는 Node.js(NestJS) + MinIO(S3 호환 스토리지) + OS를 동시에 구동할 수 없음. Node.js 단독으로도 기본 힙 메모리가 256MB~512MB를 사용하며, MinIO 최소 권장은 4GB.

3. **CPU 성능 부족** — 800MHz ARM CPU로는 HTTPS/TLS 암호화 처리, 파일 스트리밍, API 요청 처리를 동시에 수행하기 어려움. 특히 Presigned URL 서명 생성 시 암호화 연산이 필요.

4. **DSM 6.2 EOL** — Synology DSM 6.2는 보안 패치만 제공되며 신규 기능 추가가 중단됨. Container Manager, Cloudflare Tunnel 패키지 등 필수 도구가 DSM 7.x 전용.

5. **생태계 제한** — ARM 32-bit용 Docker 이미지가 극히 제한적. Node.js 공식 이미지, MinIO 공식 이미지 모두 ARM 32-bit를 지원하지 않음.

> **결론: DS215j는 현재 용도(SMB 파일 공유, 동기화 프로그램 업데이트 배포)에는 적합하나, 웹하드 백엔드 서버로는 사용할 수 없습니다. 신규 NAS 구매가 필요합니다.**

## 3. 아키텍처 옵션 비교

### 현재 아키텍처 (참고)

```
[거래처 브라우저] ──→ [Vercel: Next.js] ──→ [Railway: NestJS] ──→ Presigned URL 생성
                                                                        │
[거래처 브라우저] ←──────────── 직접 업/다운로드 ──────────────→ [Cloudflare R2]
```

- 클라이언트가 R2에 **직접** 업로드/다운로드 (Presigned URL 방식)
- NestJS는 URL 생성 + 메타데이터 관리만 담당
- 관련 코드: `StorageService` (백엔드 1개 파일) + `src/lib/r2/` (프론트 5개 파일)

### 옵션 A: 풀 온프레미스 (NestJS + MinIO on NAS)

```
[거래처 브라우저] ──→ [Vercel: Next.js] ──→ [NAS: NestJS Docker] ──→ Presigned URL 생성
                                                    │
[거래처 브라우저] ←───── Cloudflare Tunnel ─────→ [NAS: MinIO Docker]
                          (직접 업/다운로드)
```

**구성:**

- NAS에서 Docker로 NestJS + MinIO 실행
- MinIO: S3 호환 오브젝트 스토리지 (R2 대체)
- Cloudflare Tunnel로 NAS를 외부에 노출 (포트포워딩 불필요, 보안 우수)
- Next.js 프론트엔드는 Vercel에 유지 (무료 플랜)
- PostgreSQL DB는 기존 클라우드 유지 (Supabase/Railway)

**코드 변경 범위:**

- `StorageService`: S3 endpoint를 R2 → MinIO로 변경 (엔드포인트 URL, 인증 정보만 수정)
- `src/lib/r2/client.ts`: S3 endpoint 변경
- Presigned URL 패턴 그대로 유지 → **최소 코드 변경**

### 옵션 B: 하이브리드 (Railway 유지 + NAS는 스토리지만)

```
[거래처 브라우저] ──→ [Vercel: Next.js] ──→ [Railway: NestJS] ──→ Presigned URL 생성
                                                                        │
[거래처 브라우저] ←────── Cloudflare Tunnel ──────→ [NAS: MinIO Docker]
                           (직접 업/다운로드)
```

**구성:**

- NestJS는 Railway에 유지 (기존 그대로)
- NAS에는 MinIO + Cloudflare Tunnel만 구동
- Railway의 NestJS가 MinIO 엔드포인트(Tunnel URL)로 Presigned URL 생성
- 클라이언트는 Tunnel을 통해 MinIO에 직접 업/다운로드

**코드 변경 범위:**

- `StorageService`: S3 endpoint를 R2 → MinIO Tunnel URL로 변경
- 나머지 동일 → **최소 코드 변경**

### 비교표

| 평가 기준          | 옵션 A (풀 온프레미스)              | 옵션 B (하이브리드)                    |
| ------------------ | ----------------------------------- | -------------------------------------- |
| **코드 변경량**    | S3 엔드포인트 변경만 (최소)         | S3 엔드포인트 변경만 (최소)            |
| **월간 비용**      | NAS 전기료 ~1-2만원 + 인터넷        | Railway ~$5-20/월 + NAS 전기료         |
| **데이터 주권**    | 완전 사내 보관                      | 파일은 사내, 메타데이터/API는 클라우드 |
| **외부 접근 속도** | 100Mbps 업로드 (최대 ~12.5MB/s)     | 동일 (NAS 업로드 대역폭 제한)          |
| **내부 접근 속도** | LAN 직접 접근 가능 (1Gbps)          | Tunnel 경유 (약간의 레이턴시)          |
| **운영 복잡도**    | 높음 (NestJS + MinIO + Tunnel 관리) | 중간 (MinIO + Tunnel만 관리)           |
| **장애 영향**      | NAS 장애 = 웹하드 + API 전체 중단   | NAS 장애 = 파일만 중단, API는 유지     |
| **스케일링**       | NAS 하드웨어 한계                   | Railway 자동 스케일링                  |
| **백업/복구**      | NAS RAID + 외부 백업 필요           | 파일만 NAS 백업, API는 Railway 관리    |
| **보안**           | Cloudflare Tunnel (IP 노출 없음)    | 동일                                   |

### 필수 조건 충족 평가

| 필수 조건                   | 옵션 A                                                                                                                                                                                                                                                   | 옵션 B                                                                                                                        |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **R1: 체감 속도 동일**      | **조건부 충족** — 현재 R2는 Cloudflare CDN(글로벌 엣지)으로 다운로드 속도가 매우 빠름. NAS는 사무실 100Mbps 업로드가 병목. 그러나 DXF/PDF 파일이 대부분 수 MB 이하이므로 체감 차이는 크지 않을 수 있음. 대용량 파일 동시 다운로드 시에는 체감 저하 발생. | **동일** — 파일 경로는 동일하게 NAS를 경유                                                                                    |
| **R2: 24/7 무중단**         | **조건부 충족** — UPS(무정전전원장치) + 안정적 인터넷 필수. 사무실 정전이나 인터넷 장애 시 웹하드 접근 불가. Cloudflare Tunnel은 재연결 자동 처리되나, NAS 자체 장애 시 복구까지 다운타임 발생.                                                          | **부분 충족** — API는 Railway(클라우드)에서 유지되어 파일 외 기능은 정상. 파일 접근만 NAS 가용성에 의존. 장애 범위가 더 좁음. |
| **R3: 무중단 마이그레이션** | **충족 가능** — 1단계: NAS 세팅 + MinIO 구동, 2단계: R2 데이터를 MinIO로 복사(rclone), 3단계: DNS/엔드포인트 전환. R2와 MinIO를 동시 운영하면서 점진적 전환 가능.                                                                                        | **충족 가능** — 동일한 마이그레이션 전략 적용 가능                                                                            |

## 5. 최종 권고

### 권고안: 옵션 B (하이브리드) + DS723+

**이유:**

1. **장애 격리** — NAS 장애 시 파일 접근만 중단되고, API/웹사이트는 Railway에서 정상 운영. 24/7 무중단 요구사항에 더 부합.

2. **운영 부담 최소화** — NAS에서는 MinIO + Cloudflare Tunnel만 관리. NestJS 백엔드는 Railway가 자동 관리 (배포, 스케일링, 모니터링).

3. **최소 코드 변경** — `StorageService`의 S3 엔드포인트 URL과 인증 정보만 R2 → MinIO(Tunnel URL)로 변경. Presigned URL 패턴이 그대로 유지되어 프론트엔드 변경도 최소.

4. **무중단 마이그레이션 가능** — R2와 MinIO를 동시 운영하면서 `rclone`으로 데이터 복사 후, 엔드포인트만 전환.

5. **비용 효과** — Railway 비용($5~20/월)은 유지되지만, R2 스토리지/이그레스 비용 제거. NAS 초기 투자(~100만원) 후 월간 전기료만 발생.

6. **점진적 전환 가능** — 향후 NestJS도 NAS로 이전(옵션 A)하는 것은 Docker 이미지 재배포로 가능. 현 단계에서는 리스크가 낮은 하이브리드부터 시작.

### 예상 총 비용

| 항목                 | 비용                            |
| -------------------- | ------------------------------- |
| Synology DS723+ 본체 | ~60만원                         |
| RAM 8GB DDR4 ECC     | ~5만원                          |
| HDD 4TB x 2 (RAID 1) | ~20만원                         |
| UPS                  | ~15만원                         |
| **초기 투자 합계**   | **~100만원**                    |
| **월간 운영비**      | ~2만원 (전기료) + Railway $5~20 |

### 주의사항

1. **100Mbps 업로드 병목** — 현재 R2는 Cloudflare CDN을 통해 글로벌 엣지에서 서비스되어 다운로드가 매우 빠름. NAS로 전환하면 모든 외부 트래픽이 사무실 100Mbps 업로드를 경유. DXF/PDF가 소형 파일이라 대부분의 경우 체감 차이는 크지 않으나, **여러 거래처가 동시에 대용량 파일을 다운로드하면 속도 저하 발생**.

2. **24/7 가동의 현실** — 온프레미스 NAS는 본질적으로 클라우드보다 가용성이 낮음. UPS로 단기 정전은 커버되나, 장시간 정전이나 인터넷 회선 장애 시에는 서비스 중단 불가피. 이 리스크를 수용할 수 있는지 재검토 필요.

3. **백업 전략** — NAS RAID 1은 디스크 장애만 보호. 랜섬웨어, 실수로 인한 삭제 등에는 **별도 외부 백업**(Synology Hyper Backup → 클라우드 또는 USB 외장)이 필수.

4. **기존 DS215j** — 웹하드 서버로는 불가하나, 현재 용도(DXF 파일 공유, 동기화 프로그램 업데이트 배포)에는 계속 사용 가능. 교체 대상이 아님.

## 부록: 마이그레이션 개략 절차

> 이 보고서는 추천 보고서만 포함하며, 실제 구현 계획은 별도 작성 필요.

1. **NAS 구매 및 세팅** — DS723+ + RAM 증설 + HDD + UPS
2. **Docker 환경 구성** — DSM Container Manager 설치
3. **MinIO 컨테이너 배포** — S3 호환 스토리지 구동
4. **Cloudflare Tunnel 설정** — MinIO 외부 노출 (예: `files.yjlaser.com`)
5. **데이터 마이그레이션** — `rclone`으로 R2 → MinIO 데이터 동기화
6. **엔드포인트 전환** — NestJS `StorageService` S3 엔드포인트 변경
7. **검증** — 업/다운로드, 멀티파트, 삭제 기능 테스트
8. **R2 정리** — 안정화 후 R2 데이터 삭제
