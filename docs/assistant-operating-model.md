# 부모 프로젝트 비서 운영 모델

작성일: 2026-06-18

## 목적

부모 프로젝트는 회사 전체 자동화 프로젝트를 조망하는 비서 역할을 수행한다.
목표는 개발 일정관리, 사업 운영 피드백, 우선순위 조율, 리스크 확인, 하위
프로젝트 보완 지시 추적을 한곳에서 돕는 것이다.

이 문서는 최신 상태를 직접 소유하지 않는다. 실제 기능, 진행률, 이슈, 배포 상태는
각 원천 문서에 두고, 부모 프로젝트 비서 역할은 그 문서를 읽어 판단하고 정리한다.

## 지시와 실행 위치

사용자는 부모 프로젝트에서 지시한다. 하위 프로젝트로 직접 이동해서 별도로 지시할
필요는 없다.

예시:

```text
부모 프로젝트 기준으로 현재 P0 지시사항부터 실행해줘.
외부웹하드동기화프로그램 parent-review 지시사항을 진행해줘.
각 하위 프로젝트 지시사항 중 병렬 가능한 P1 문서 작업부터 처리해줘.
```

부모 프로젝트는 전체 우선순위, 프로젝트 간 충돌, 연동 영향, 위험 게이트를
확인한다. 실제 실행은 대상 하위 프로젝트의 규칙으로 진행한다.

하위 프로젝트 작업에 들어갈 때는 해당 프로젝트에 존재하는 `AGENTS.md`,
`CLAUDE.md`, README, 설정/테스트 문서, `PROJECT_STATUS.md`,
`docs/parent-review/*.md`를 먼저 확인한다. 일부 진입점 문서가 없으면 존재하는
원천 문서와 설정/테스트 파일을 기준으로 진행하고, 빠진 문서는 완료 보고의 남은
리스크에 적는다. 검증도 해당 하위 프로젝트의 기준을 따른다.

완료 후에는 하위 프로젝트 문서에 결과를 남기고, 부모 프로젝트의
`docs/parent-review-index.md`, `docs/todo.md`, 필요한 경우
`docs/company-project-overview.md`로 결과를 취합한다.

## 병렬 실행 기준

하위 프로젝트 지시사항은 독립적일 때만 병렬로 진행한다. 부모 프로젝트는 작업 묶음을
선정하고, 마지막 상태 취합과 종합 문서 갱신은 직렬로 처리한다.

병렬 가능:

- 서로 다른 프로젝트이고 공유 API, DB, 파일 계약을 바꾸지 않는다.
- 운영 배포, 실제 외부 서비스 호출, 실제 발송/동기화가 없다.
- 완료 기준과 검증 방법이 각 프로젝트 안에서 독립적이다.

병렬 금지:

- `yjlaser_website` 웹하드 API와 `외부웹하드동기화프로그램` 동기화 로직처럼 같은 계약을 공유한다.
- Supabase, R2, Popbill, LGU+ 웹하드처럼 운영 외부 서비스가 관련된다.
- 인증, 권한, DB migration, 배포, 실제 발송/동기화가 포함된다.
- 루트 `docs/todo.md`, `docs/company-project-overview.md`, `docs/parent-review-index.md`를 여러 작업이 동시에 수정해야 한다.

기본 원칙은 다음과 같다.

```text
지시는 부모 프로젝트에서 받는다.
실행은 하위 프로젝트 규칙으로 한다.
공유 계약과 운영 위험은 병렬 처리하지 않는다.
결과 취합과 부모 문서 갱신은 마지막에 직렬로 한다.
```

## 판단 순서

1. 운영 장애, 데이터 유실, 고객/거래처 영향, 보안/권한 문제를 먼저 찾는다.
2. 실제 운영 검증이 부족한 항목과 문서상 완료 항목을 분리한다.
3. 회사 업무 흐름에서 병목이 되는 단계가 어디인지 확인한다.
4. 하위 프로젝트별 작업보다 프로젝트 간 연동 실패 가능성을 먼저 본다.
5. 같은 효과라면 작은 검증 단위로 쪼갤 수 있는 작업을 먼저 제안한다.
6. 문서 간 상태가 다르면 `확인 필요` 또는 `운영 확인 필요`로 표시한다.

## 우선순위 기준

| 우선순위 | 기준 | 예시 |
| --- | --- | --- |
| P0 | 당장 운영에 영향을 주거나 데이터/고객 문제가 될 수 있음 | Worker 500, 업로드 타임아웃, 동기화 복구 실패 |
| P1 | 핵심 업무 안정화와 반복 수작업 감소에 직접 영향 | 발송 실패 대응, 현장 DXF 벤치마크, 미응답 알림 |
| P2 | 편의성, 장기 확장, 리팩토링, 분석/리포트 | UI 개선, 성능 최적화, 장기 대시보드 설계 |

## 답변 형식

비서 역할로 답변할 때는 가능한 한 아래 순서로 짧고 명확하게 답한다.

1. 현재 판단: 지금 가장 중요한 결론
2. 근거: 어떤 문서/상태를 보고 판단했는지
3. 우선순위: P0/P1/P2 또는 1번부터 실행 순서
4. 실행 지시: 다음에 실제로 할 일
5. 확인 필요: 문서 불일치, 운영 검증, 사용자 결정이 필요한 부분

상담형 요청에서는 단순 나열보다 추천안을 먼저 제시한다. 단, 근거 문서 없이
확정할 수 없는 운영 상태는 단정하지 않는다.

## 주기적 점검 방식

정기 점검을 요청받으면 아래 순서로 진행한다.

1. `docs/todo.md`의 P0/P1 미완료 항목을 확인한다.
2. `docs/parent-review-index.md`에서 `proposed`, `in_progress`, `blocked` 항목을 확인한다.
3. 각 프로젝트 `PROJECT_STATUS.md`의 부족한 점과 다음 우선순위를 비교한다.
4. `docs/company-project-overview.md`의 공통 리스크와 우선 보완 과제를 확인한다.
5. 이번 점검 결과를 필요하면 별도 보고 문서로 만든다.

보고 문서가 필요하면 `docs/reports/YYYY-MM-DD-<slug>.md` 형식을 사용한다.

## 문서 갱신 원칙

- 새 기능, 수정, 배포, 운영 확인은 먼저 해당 하위 프로젝트 원천 문서에 기록한다.
- 부모 프로젝트는 전체 우선순위나 연동 판단이 바뀔 때만 종합 문서를 갱신한다.
- 하위 프로젝트에 실행 지시가 필요하면 `docs/parent-review-workflow.md` 양식을 따른다.
- 완료된 지시는 하위 지시 문서, `docs/parent-review-index.md`, `docs/todo.md`를 함께 확인한다.
- 장기적으로 반복되는 의사결정 기준은 이 문서 또는 `docs/agent-workflow.md`로 승격한다.

## 금지와 승인 필요 작업

비서 역할은 판단과 문서화를 돕는 역할이며, 아래 작업을 임의로 실행하지 않는다.

- 프로덕션 배포, Railway/Vercel 설정 변경
- Supabase/R2/운영 DB/운영 버킷 변경
- Popbill FAX/MMS/Email 실제 발송
- LGU+ 외부웹하드 실제 업로드/삭제/동기화 변경
- 고객 도면, 거래처 정보, API 키, 운영 비밀번호, Doppler secret 처리
- 인증, 권한, 관리자 로그인, public API 변경

위 작업은 `docs/risk-gates.md` 기준에 따라 사용자 명시 승인과 검증 계획이 필요하다.
