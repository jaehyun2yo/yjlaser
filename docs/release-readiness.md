## 목적

`docs/features-list.md` 기준 release train 항목을 실제 배포 전후 확인 가능한 checklist로 분해한다. 최초 REL-001 범위는 29개 `implemented (..., pending deploy)`였고, 2026-05-13 남은 작업 처리 중 `worker-pin-brute-force-rate-limit`, `webhard-classify-failed-notification-dedupe` 2개가 추가되어 현재 추적 대상은 31개다. 2026-05-14 기준 2개는 backend production 배포가 끝났지만 수동 QA가 남아 `deployed (..., needs QA)` 상태다.

## 공통 배포 전제

- Frontend 배포 대상은 Vercel, backend 배포 대상은 Railway `webhard-api`다.
- Railway backend는 Node 20 환경에서 `pnpm@10.23.0` 고정 Docker build가 성공해야 한다.
- 운영 DB 작업 전 `DATABASE_URL` / `DIRECT_URL` 대상과 backup/rollback 경로를 확인한다.
- Cloudflare R2 credentials, bucket, CORS, presigned URL 동작을 업로드/다운로드 smoke에서 확인한다.
- 외부 프로그램 연동은 실제 운영 또는 staging API base URL과 `X-API-Key` 스코프를 확인한다.
- 수동 QA 증거는 화면 캡처, contact id, file id, folder id, sync log id, timestamp note 중 하나 이상으로 남긴다.

## 최종 추천 실행 순서

1. **Release baseline 확인**: Vercel/Railway 현재 배포 commit, Railway Docker `pnpm@10.23.0`, backend/frontend build 경로를 먼저 확인한다.
2. **P0 의존성 처리**: `password_reset_tokens` migration 대상 DB, backup/rollback, SMTP, site URL, backup API key scope, realtime session cookie domain을 확정한다.
3. **Backend 먼저 배포**: Railway backend에 Webhard, Worker realtime, integration, backup, monitoring API 변경을 반영한다. API가 먼저 준비되어야 Vercel UI가 새 endpoint와 response shape를 사용할 수 있다.
4. **Frontend 배포**: Vercel frontend에 Worker, Webhard, Company/Auth, Admin dashboard, Next proxy 변경을 반영한다.
5. **외부 프로그램 smoke**: 외부웹하드 sync batch upload와 레이저 완료 API를 실제 또는 staging API key로 확인한다.
6. **Webhard E2E QA**: admin/company actor로 upload, download, rename, move, DXF preview, badge refresh, lazy folder, session expiry, cookie ByteString 경로를 한 번에 확인한다.
7. **Worker/mobile QA**: Worker 새 문의 알림, 최신 도면 다운로드, 추가 도면 업로드, 납품증빙 모바일 camera, `문의/완료/` 이동을 확인한다.
8. **Inquiry/Company Auth QA**: 공개 문의 제출, 비밀번호 reset email 왕복, 승인 대기 로그인 메시지를 확인한다.
9. **Admin/Monitoring QA**: backup 권한 matrix, activity range, storage breakdown, pipeline backlog 민감정보 마스킹을 확인한다.
10. **Design/System smoke**: light/dark visual smoke와 changed-file static gate를 확인한다.
11. **상태 갱신**: 각 feature를 `shipped`, `needs hotfix`, `deferred` 중 하나로 결정하고 `docs/features-list.md`와 `docs/task-board.md`를 갱신한다.
