# AI 작업 워크플로우 어댑터

YJLaser는 여러 하위 프로젝트를 가진 업무 자동화 모노레포다. 글로벌
`global-orchestrator`가 활성화되어 있으면 공통 절차를 담당하고, 이 문서는
YJLaser에서 어떤 리뷰와 검증을 선택할지 정한다. 글로벌 오케스트레이터가
비활성화된 세션에서는 이 문서의 절차를 수동 기준으로 적용한다.

## 기본 흐름

사용자가 작업을 요청하면 기본적으로 계획 우선 흐름으로 동작한다. 글로벌
`global-orchestrator`가 활성화되어 있으면 해당 하네스를 사용하고, 꺼져 있으면
아래 절차를 수동으로 따른다. 비 trivial 작업은 구현 전에 Plan mode에 해당하는
명확화, 계획 수립, 계획 리뷰 게이트를 먼저 거친다. 다만 안전하고 국소적인
가벼운 수정은 예외로 분류해 바로 수행할 수 있다.

```text
요청 접수
→ 프롬프트 품질/모호성/위험도 확인
→ 가벼운 수정 여부와 `간단한` 지시어 확인
→ 비 trivial 작업은 Plan mode 상당의 브레인스토밍/계획 단계 진입
→ 계획 수립
→ GStack 계획 리뷰 추천 및 사용자 승인
→ 구현
→ 검증
→ 독립 리뷰
→ Compound/GStack learning 판단
→ 완료 보고
```

## `간단한` 지시어 분기

사용자가 작업 지시에 `간단한`을 붙이면 먼저 간단 작업 후보로 분류한다.
이 분기는 반복적인 승인 질문을 줄이기 위한 편의 신호이며, 위험 게이트를
우회하는 권한이 아니다.

간단 작업으로 바로 수행할 수 있는 예:

- 안전한 로컬 문서 수정
- 명백한 오타, 문구, 주석 수정
- 기존 패턴이 분명한 작은 코드 수정
- 단일 명령 출력 확인 또는 간단한 파일 조회

아래 조건 중 하나라도 있으면 `간단한`이 붙어도 즉시 수행하지 않는다.

- 프로덕션 배포, DB, 마이그레이션, 외부 서비스 실제 호출
- 인증, 권한, 관리자 로그인, public API 변경
- 고객 도면, 거래처 정보, API 키, Doppler secret 등 민감 정보 처리
- 삭제, 동기화, 실제 발송, 업로드처럼 되돌리기 어려운 부작용
- 목표, 대상, 성공 기준, 검증 방법이 불명확한 작업

`간단한`이 없더라도 가벼운 수정으로 판단되면 바로 수행한다. 가벼운 수정은
아니지만 작은 작업으로 판단되는 경우에는 fast path를 제안하고 사용자 승인을
받는다.

## 브레인스토밍 강화 기준

브레인스토밍은 오래 토론하기 위한 단계가 아니라, 구현에 들어가기 전에
작업 프롬프트를 충분히 명확하게 만드는 단계다. 비 trivial 작업에서는 아래
항목을 빠르게 채운 뒤 계획으로 넘어간다.

- 목표: 사용자가 얻고 싶은 결과
- 대상: 앱, 파일, 워크플로우, 기능, 명령
- 성공 기준: 완료 여부를 판단할 관찰 가능한 조건
- 제약: 기술, UX, 호환성, 일정, 스타일, 안전 기준
- 비목표: 이번 작업에서 바꾸지 않을 것
- 위험: 데이터, 인증, 외부 시스템, 배포, 비용, 메시지 발송
- 검증: 테스트, 빌드, 브라우저 QA, 수동 확인, 명령 출력

핵심 항목이 비어 있으면 추측해서 구현하지 않고 한 번에 하나의 집중 질문을
한다. 질문은 가능하면 선택지를 제공하고, 답변이 모이면 작업 지시문으로
재정리한 뒤 계획을 세운다.

## 계획 리뷰 추천 매핑

계획이 필요한 작업은 구현 전에 아래 기준으로 리뷰를 추천한다.

| 작업 유형 | 추천 리뷰 | 기준 |
|---|---|---|
| 일반 코드, 백엔드, 아키텍처, 리팩토링 | `gstack-plan-eng-review` | 구조, 테스트, 데이터 흐름, 구현 가능성이 핵심일 때 |
| 웹 UI, 화면 흐름, 사용자 경험 | `gstack-plan-design-review` | 레이아웃, 상호작용, 시각 품질, 사용성 판단이 있을 때 |
| 설치, 개발환경, CLI, 테스트/CI, 배포 절차 | `gstack-plan-devex-review` | 다른 PC/개발자가 재현해야 하는 절차일 때 |
| 제품 범위, 우선순위, 업무 가치, 단계 나누기 | `gstack-plan-ceo-review` | 무엇을 먼저 만들지 또는 범위가 적절한지 판단해야 할 때 |
| 여러 하위 프로젝트, DB/보안/배포 포함, 큰 기능 | `gstack-autoplan` | 여러 관점 리뷰가 필요한 작업일 때 |

사용자에게는 아래 형식으로 묻는다.

```text
추천하는 계획 리뷰는 `<skill>`입니다. 이유는 <reason>입니다.
이 리뷰를 먼저 진행할까요?
```

## 작업 유형별 기본 경로

| 작업 | 기본 경로 |
|---|---|
| 새 기능/동작 변경 | 브레인스토밍 → 계획 → 계획 리뷰 추천 → 구현 |
| 버그 수정 | 문제 재현/디버깅 → 테스트 계획 → 구현 → 검증 |
| UI 변경 | 브레인스토밍 → `gstack-plan-design-review` 추천 → 구현 → 브라우저 QA |
| DB/Prisma 변경 | 브레인스토밍 → `gstack-plan-eng-review` 또는 `gstack-autoplan` 추천 → 마이그레이션 안전성 검토 |
| 인증/API 변경 | 브레인스토밍 → 보안/API 계약 리뷰 포함 → 테스트 |
| 배포/운영 변경 | 위험 게이트 → `gstack-plan-devex-review` 또는 deployment review |
| `간단한`이 붙은 안전한 작은 작업 | 즉시 수행 → 검증 → 짧은 리뷰 |
| 가벼운 수정으로 판단되는 안전한 작은 작업 | 즉시 수행 → 검증 → 짧은 리뷰 |
| `간단한`이 없는 문서만 수정 | 가벼운 수정이면 즉시 수행, 범위/영향 판단이 필요한 경우 fast path 승인 |

## 완료 보고 형식

최종 보고에는 다음을 포함한다.

- 변경 파일
- 실행한 검증과 결과
- 실행하지 못한 검증과 이유
- 리뷰 방식: 별도 reviewer agent, GStack/Compound 리뷰, 또는 fresh-context 리뷰
- 남은 리스크
- learning/artifact 기록 여부
